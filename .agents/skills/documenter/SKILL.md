# Скилл: Документовед (Documenter)

## Описание

Аудит и синхронизация кода с документацией. Сравнивает реальный код с тем, что описано в `docs/`, находит расхождения, дописывает недостающее. Работает по конкретной области, папке, модулю или после изменений.

## Ключевые слова активации

- «задокументируй», «обнови документацию», «допиши в доку»
- «аудит документации», «проверь документацию»
- «что не задокументировано», «где пробелы в доке»
- «документация устарела», «синхронизируй доку с кодом»
- «опиши этот модуль», «задокументируй папку»

## Структура документации проекта

```
docs/
  README.md                    ← Навигационный хаб
  architecture/                ← 6 файлов: общая архитектура
  backend/                     ← ~30+ файлов: бэкенд (фичи, CRUD, сервисы, callback)
    features/                  ← 18 подпапок по доменам
    core_services/             ← VK Service Facade
    crud/                      ← CRUD обзор
    vk_callback/               ← 7 файлов
  frontend/                    ← ~30+ файлов: фронтенд (фичи, shared, хуки)
    features/                  ← 13 подпапок по доменам
    shared/                    ← 4 файла (API, хуки, компоненты)
  api/                         ← 3 файла: контракт, схема БД, аудит VK
  ui-kit/                      ← 15 файлов: дизайн-система + 9 компонентов
  deployment/                  ← 3 файла: деплой
  changelog/                   ← 19 файлов: история изменений
  sandbox/                     ← 11 файлов: тесты VK API
  training/                    ← 5 файлов: учебный центр
  misc/                        ← 6 файлов: TODO, карта, спеки
```

### Важно: формат файлов

- **Фронтенд-документация:** обычный `.md`
- **Бэкенд-документация:** `.md.txt` (историческое правило для Google AI Studio)
- Исключения: `README.md` в корне папок — без `.txt`

## Процедура 1: Аудит конкретной папки / модуля

**Триггер:** «проверь документацию для X», «аудит доки папки Y», «что не задокументировано в Z»

### Алгоритм

```
1. ОПРЕДЕЛИТЬ ОБЛАСТЬ
   - Пользователь указал папку/модуль? → использовать
   - Нет → спросить что проверять

2. СОБРАТЬ КОД
   - Прочитать все файлы в целевой папке
   - Выписать: модули, классы, функции, эндпоинты, хуки, компоненты
   - Зафиксировать: параметры, типы, зависимости

3. НАЙТИ СУЩЕСТВУЮЩУЮ ДОКУМЕНТАЦИЮ
   - Маппинг папка кода → папка документации (см. references/doc_map.md)
   - Прочитать все связанные doc-файлы

4. СРАВНИТЬ
   Для каждого элемента кода проверить:
   ┌──────────────────────┬─────────────────────────────────┐
   │ Проверка             │ Статус                          │
   ├──────────────────────┼─────────────────────────────────┤
   │ Упомянут в доке?     │ ✅ Да / ❌ Нет                  │
   │ Описание актуально?  │ ✅ Совпадает / ⚠️ Устарело      │
   │ Параметры/типы верны?│ ✅ Да / ⚠️ Расхождение          │
   │ Новые поля/методы?   │ ✅ Описаны / ❌ Не описаны      │
   └──────────────────────┴─────────────────────────────────┘

5. ВЫВЕСТИ ОТЧЁТ АУДИТА
   (формат — см. ниже)

6. СПРОСИТЬ: «Дописать недостающее?»
   - Да → дописать в существующие файлы
   - Нет → оставить отчёт как есть
```

### Формат отчёта аудита

```
═══════════════════════════════════════
  АУДИТ ДОКУМЕНТАЦИИ: <область>
═══════════════════════════════════════

📁 Папка кода: backend_python/services/automations/
📄 Документация: docs/backend/features/automations/

ПОКРЫТИЕ: 7 из 12 файлов задокументированы (58%)

✅ ЗАДОКУМЕНТИРОВАНО:
  • AutomationOrchestrator — docs/backend/features/automations/01_OVERVIEW.md
  • stories_publisher.py — описан в 02_STORIES.md
  • ...

❌ НЕ ЗАДОКУМЕНТИРОВАНО:
  1. viewers_details.py — нет упоминания, НОВЫЙ ФАЙЛ
     → Содержит: класс ViewersDetailsService, 3 метода
     → Рекомендация: добавить в 02_STORIES.md или создать отдельный файл
     
  2. bulk_stories.py — упомянут, но описание устарело
     → В доке: "принимает list[int]"
     → В коде: "принимает StoryBulkRequest (pydantic)"
     → Рекомендация: обновить описание параметров

⚠️ УСТАРЕВШЕЕ:
  • В доке упомянут файл scheduler_v1.py — в коде его нет (удалён?)

═══════════════════════════════════════
  ВЕРДИКТ: Требуется обновление (4 пункта)
═══════════════════════════════════════
```

## Процедура 2: Документирование после изменений

**Триггер:** «задокументируй что изменилось», «обнови доку после рефакторинга»

### Алгоритм

```
1. УЗНАТЬ ЧТО ИЗМЕНИЛОСЬ
   Варианты:
   a. git diff (если есть коммиты) → get_changed_files
   b. Пользователь описал → конкретные файлы
   c. Контекст диалога → что делали в этой сессии

2. ДЛЯ КАЖДОГО ИЗМЕНЁННОГО ФАЙЛА
   a. Прочитать новый код
   b. Найти соответствующую документацию (doc_map)
   c. Определить тип изменения:
      - Новый файл → создать документацию
      - Изменён существующий → обновить документацию
      - Удалён → удалить из документации
      - Переименован → обновить пути

3. ОБНОВИТЬ ДОКУМЕНТАЦИЮ
   - Дописать в существующий файл (если контекст)
   - Или создать новый (если новый модуль)
   - Обновить DOCUMENTATION_MAP если были структурные изменения

4. ПОДТВЕРДИТЬ
   Кратко показать что обновлено
```

## Процедура 3: Документирование конкретного функционала

**Триггер:** «опиши функционал X», «задокументируй как работает Y»

### Алгоритм

```
1. НАЙТИ КОД ФУНКЦИОНАЛА
   - По описанию пользователя → grep/semantic_search
   - Найти все связанные файлы:
     • Роутер (backend_python/routers/)
     • Сервис (backend_python/services/)
     • CRUD (backend_python/crud/)
     • Схемы (backend_python/schemas/)
     • Фронтенд-компоненты (features/)
     • Хуки (features/*/hooks/ или hooks/)

2. ПРОАНАЛИЗИРОВАТЬ
   - Назначение
   - Входные/выходные данные
   - Эндпоинты API (метод, путь, параметры, ответ)
   - Зависимости
   - Бизнес-логика (основные шаги)
   - Особенности / edge cases

3. НАПИСАТЬ ДОКУМЕНТАЦИЮ
   Формат зависит от типа:
   - Бэкенд-фича → references/backend_feature_template.md
   - Фронтенд-фича → references/frontend_feature_template.md
   - API-эндпоинт → references/api_template.md

4. РАЗМЕСТИТЬ
   - Определить куда (по doc_map)
   - Дописать в существующий файл или создать новый
   - Обновить оглавление (если есть)
```

## Процедура 4: Полный аудит области

**Триггер:** «полный аудит документации», «проверь всю доку бэкенда/фронтенда»

### Алгоритм

```
1. Определить область (бэкенд / фронтенд / всё)
2. Собрать список всех файлов кода в области
3. Собрать список всех файлов документации
4. Построить матрицу покрытия:

   Код                          → Документация
   ─────────────────────────────────────────────
   routers/posts.py             → docs/backend/features/posts/
   routers/market.py            → docs/backend/features/market/
   features/posts/              → docs/frontend/features/post_modal/
   ...

5. Для каждой пары: быстрая проверка (файл есть? актуален ли?)
6. Вывести сводную таблицу покрытия
7. Предложить план обновления (по приоритету)
```

---

## Правила

1. **Не переписывать существующую доку** — только дополнять и обновлять
2. **Сохранять стиль** — читать существующий файл, писать в том же стиле
3. **Бэкенд-файлы — `.md.txt`** (историческое правило)
4. **Фронтенд-файлы — `.md`**
5. **Не документировать очевидное** — фокус на бизнес-логике, edge cases, неочевидных решениях
6. **Спрашивать перед созданием** нового файла документации — может, лучше дописать в существующий
7. **Обновлять карты** — `DOCUMENTATION_MAP.md` / `DOCUMENTATION_MAP.md.txt` если меняется структура

## Делегирование саб-агентам

Следующие шаги рекомендуется выполнять через `runSubagent` для экономии контекста:

| Шаг | Что делегировать | Промпт для саб-агента |
|---|---|---|
| ⚡ Сбор кода | Анализ всех файлов модуля (10–20 файлов) | «Прочитай все файлы в {backend_python/routers,services,crud,schemas}/{модуль}. Для каждого верни: функции с docstring, типы параметров и возвратов, комментарии, TODO. Формат — JSON» |
| ⚡ Сбор документации | Чтение существующей документации | «Прочитай все файлы в docs/{area}/. Верни: заголовки секций, описанные модули, устаревшие ссылки, пустые секции. Формат — JSON» |
| Матрица покрытия | Сопоставление кода ↔ доки | «Вот код: {JSON кода}. Вот дока: {JSON доки}. Построй матрицу: функция → задокументирована (да/нет). Верни список недокументированных функций» |
| Генерация шаблонов | Создание документации по шаблону | «На основе данных {JSON кода} и шаблона из references/{шаблон}.md сгенерируй полный .md файл документации. Верни готовый текст» |

**⚡ ПАРАЛЛЕЛЬНО:** «Сбор кода» + «Сбор документации» — запускать одновременно (один читает исходники, другой — существующую доку). «Матрица покрытия» — после обоих.
