# Release — Ведение журнала релизов

## Файловая структура

```
docs/changelog/
  release.md          ← Технический журнал (том 1)
  release2.md         ← Технический журнал (том 2)
  ...
  user_rlease.md      ← Пользовательский журнал (том 1)
  user_rlease2.md     ← Пользовательский журнал (том 2)
  ...

features/updates/data/
  updatesData.ts      ← Главный файл: интерфейсы + массив updatesData[]
  release2Data.ts     ← Данные релиза 2
  release3Data.ts     ← Данные релиза 3
  ...
```

## Критическое правило: определение актуального файла

**Перед ЛЮБОЙ записью** обязательно:
1. Просканировать `docs/changelog/` — найти ВСЕ файлы `release*.md` и `user_rlease*.md`
2. Определить файл с **наибольшим номером** — это актуальный файл
   - `release.md` = номер 1, `release2.md` = номер 2, и т.д.
3. Записывать **ТОЛЬКО** в актуальный файл
4. Если > 50 записей — создать следующий по номеру файл (парой: technical + user)

## Процедуры

### «Новый релиз» / «в новый релиз»

Триггеры: «новый релиз», «в новый релиз», «начни новый релиз», «создай релиз», «открой новый релиз»

> **КРИТИЧЕСКОЕ ПРАВИЛО:** Когда пользователь пишет «в новый релиз» — это значит **СОЗДАТЬ НОВЫЕ ФАЙЛЫ**, а не писать в существующие. Каждый новый релиз = новый набор файлов + новая вкладка на фронтенде.

**Процедура:**

1. Просканировать `docs/changelog/` — найти файлы `release*.md` с наибольшим номером (N)
2. Следующий номер = N + 1
3. **Создать 3 новых файла:**
   - `docs/changelog/release{N+1}.md` — технический журнал (по шаблону из [references/templates.md](references/templates.md), секции 🆕, 🔴, ✅)
   - `docs/changelog/user_rlease{N+1}.md` — пользовательский журнал (по шаблону, секции 🆕, 🔴, ✅)
   - `features/updates/data/release{N+1}Data.ts` — данные для фронтенда (по шаблону из [references/frontend_format.md](references/frontend_format.md))
4. В `release{N+1}Data.ts`:
   - `date: 'хх.хх.хххх'` — дата заполнится при деплое фронтенда
   - `status: 'in-progress'`
   - `summary` — из контекста запроса пользователя
   - `entries: []` — пустой массив, записи добавляются по мере работы
5. **Подключить в `features/updates/data/updatesData.ts`:**
   - Добавить `import { release{N+1}Data } from './release{N+1}Data';`
   - Вставить `release{N+1}Data` **первым** элементом массива `updatesData`
6. Это создаст **новую вкладку** «Обновление от хх.хх.хххх» в сайдбаре страницы обновлений (дата выставится при деплое фронтенда)
7. Записать первую запись из контекста (баг/фича/улучшение) во все 3 файла
8. Проверить: `get_errors` на `release{N+1}Data.ts` и `updatesData.ts`

### «Запиши баг» / «добавь в релиз»

Триггеры: «релиз — запиши баг», «добавь в релиз», «запиши в релиз проблему», «релиз: [описание]»

1. Найти актуальные файлы (технический + пользовательский)
2. Извлечь суть проблемы из контекста диалога
3. В технический файл → секция `## 🔴 Открытые задачи` → добавить запись по шаблону
4. В пользовательский файл → секция `## 🔴 Открытые задачи` → добавить запись простым языком
5. **→ Фронтенд:** Добавить `UpdateEntry` с `type: 'bugfix'` в актуальный `releaseXData.ts` (см. «Автоматическое обновление фронтенда»)

Шаблоны: прочитать [references/templates.md](references/templates.md)

### «Релиз — решено» / «закрой баг»

Триггеры: «найди баг в релизе и напиши решение», «обнови релиз с решением», «закрой баг», «релиз — решено»

1. Найти актуальные файлы
2. Найти открытую задачу в `## 🔴 Открытые задачи` (если нет — проверить предыдущие тома)
3. Перенести в `## ✅ Решённые задачи`
4. Дополнить причиной и решением (технический файл — подробно, пользовательский — простым языком)
5. Убрать метку `**Статус:** 🔴 В работе`
6. **→ Фронтенд:** Обновить соответствующий `UpdateEntry` в `releaseXData.ts` — заполнить `userSolution` и `technicalDetails`

Шаблоны: прочитать [references/templates.md](references/templates.md)

### «Новая фича» / «добавили функционал»

Триггеры: «релиз — новая фича», «добавили функционал», «новая возможность», «в релиз изменения по...»

1. Найти актуальные файлы
2. Технический файл → добавить запись с техническими деталями
3. Пользовательский файл → секция `## 🆕 Новые возможности` → шаблон фичи
4. **→ Фронтенд:** Добавить `UpdateEntry` с `type: 'feature'` в актуальный `releaseXData.ts` (см. «Автоматическое обновление фронтенда»)

Шаблоны: прочитать [references/templates.md](references/templates.md)

### Автоматическое обновление фронтенда

> **КРИТИЧЕСКОЕ ПРАВИЛО:** После КАЖДОЙ записи в `releaseX.md` + `user_rleaseX.md` **ОБЯЗАТЕЛЬНО** добавить соответствующий `UpdateEntry` в `features/updates/data/releaseXData.ts`.

**Процедура:**

1. Прочитать [references/frontend_format.md](references/frontend_format.md) для формата интерфейсов
2. Определить номер актуального релиза (X) — тот же, что у releaseX.md
3. Прочитать `features/updates/data/releaseXData.ts` — найти массив `entries`
4. Добавить новый `UpdateEntry` в конец массива `entries`:
   - `id` — уникальный slug (проверить что не повторяется)
   - `type` — `'bugfix'` / `'feature'` / `'improvement'` (маппинг из секции)
   - `section` — из таблицы стандартных значений в frontend_format.md
   - `impactType` — `'visual'` / `'logic'` / `'both'` (по затронутым файлам)
   - `title` — из заголовка записи в releaseX.md
   - `userDescription` — из `**Проблема:**` в user_rleaseX.md (или `**Что добавили:**`)
   - `userSolution` — из `**Решение:**` в user_rleaseX.md (или `**Что изменилось:**`)
   - `userInstructions` — из `**Как использовать:**` (только для feature)
   - `userBenefits` — из `**Что теперь можно:**` (только для feature)
   - `technicalDetails` — краткая выжимка из технического releaseX.md
5. Проверить: `get_errors` на releaseXData.ts
6. Если файл `releaseXData.ts` **не существует** — создать его по шаблону из frontend_format.md и подключить в `updatesData.ts` (импорт + первый элемент массива)

### «Обнови фронт обновлений» / «сформируй релиз»

Триггеры: «обнови страницу обновлений», «сформируй релиз для фронта», «запиши в обновления на фронт»

> **Примечание:** Если фронтенд-файл уже обновляется автоматически (см. «Автоматическое обновление фронтенда»), эта процедура нужна только для первичного создания `releaseXData.ts` или финализации (статус `'published'`).

1. Прочитать [references/frontend_format.md](references/frontend_format.md)
2. Определить номер релиза (X)
3. Если `releaseXData.ts` **не существует** — создать файл + подключить в `updatesData.ts`
4. Если существует — убедиться что все записи из releaseX.md + user_rleaseX.md отражены в entries
5. Установить `status: 'published'` если релиз завершён
6. Проверить: `get_errors`, уникальность `id`

## Делегирование саб-агентам

Для экономии контекстного окна следующие шаги рекомендуется выполнять через `runSubagent`:

| Шаг | Что делегировать | Промпт для саб-агента |
|---|---|---|
| ⚡ Определение актуального файла | Сканирование docs/changelog/ + подсчёт записей | «Просканируй docs/changelog/. Найди файлы release*.md и user_rlease*.md с наибольшим номером. Верни: technical_file, user_file, open_count, resolved_count» |
| «Релиз — решено» | Поиск открытой задачи по нескольким томам | «В docs/changelog/ найди все release*.md. В каждом найди секцию '🔴 Открытые задачи' и запись '{описание}'. Верни: файл, текст записи, номера строк» |
| ⚡ Обновление фронтенда | Сбор контекста из reference + releaseXData.ts | «Прочитай references/frontend_format.md и features/updates/data/releaseXData.ts. Верни: интерфейс UpdateEntry, список id существующих entries, номер строки для вставки» |
| Сверка changelog ↔ фронт | Сравнение 3 файлов на полноту | «Сравни releaseX.md, user_rleaseX.md и releaseXData.ts. Верни записи, которые есть в changelog, но отсутствуют в данных фронта: [{title, type, section}]» |

**⚡ ПАРАЛЛЕЛЬНО:** «Определение актуального файла» + «Обновление фронтенда» — запускать одновременно (один сканирует changelog/, другой читает references/ + data/). «Сверка changelog ↔ фронт» — после обоих.
