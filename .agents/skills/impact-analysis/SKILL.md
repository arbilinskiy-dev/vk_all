# Impact Analysis — Анализ влияния изменений

## Когда активировать

- **Автоматически** для задач уровня **L1** и выше (см. классификацию в ai_instr)
- Задача изменяет **UI-компонент** (добавление элемента, изменение поведения)
- Задача добавляет **новый элемент в существующий интерфейс** (кнопка, колонка, поле, чекбокс)
- Задача затрагивает **API-контракт** (новый параметр, изменение response)
- Явные триггеры: «проанализируй влияние», «что затронет», «покажи варианты»

## Когда НЕ активировать

- **L0** — точечный фикс в 1 файле (typo, стиль, значение константы)
- Чисто документационные изменения
- Пользователь явно сказал «просто сделай» / «без анализа»

## Workflow (4 шага)

### Шаг 1: Карта связанных компонентов

> ⚡ Делегировать саб-агенту `Explore` если файлов > 5

Найти **все файлы**, которые затрагивает задача:

```
Бэкенд:
  - Роутер (routers/)
  - Сервис (services/)
  - CRUD (crud/)
  - Схема Pydantic (schemas/)
  - Модель ORM (models_library/)
  - Тесты (tests/)

Фронтенд:
  - Компонент UI (features/{модуль}/components/)
  - Хук логики (features/{модуль}/hooks/)
  - API-модуль (services/api/)
  - Типы (features/{модуль}/types.ts или shared/types/)
  - Тесты (tests/{модуль}/)
```

**Выход:** список файлов + граф зависимостей (кто импортирует что).

### Шаг 2: UI Discovery (ключевой шаг)

> Этот шаг предотвращает промахи вида «добавил чекбокс вместо элемента списка»

Если задача затрагивает фронтенд:

1. **Открой целевой компонент** и прочитай его ПОЛНОСТЬЮ
2. **Определи существующие UI-паттерны:**
   - Как организованы списки? (чекбоксы, карточки, строки таблицы)
   - Как добавляются новые элементы? (кнопка «Добавить», API загрузка)
   - Какие пропсы принимает компонент? Какие данные доступны?
3. **Найди аналогичный реализованный функционал** в проекте:
   - Есть ли в другом модуле похожая задача, которая уже решена?
   - Как она решена? (паттерн-образец)
4. **Если задача "добавить X в Y"** — проверь:
   - Нет ли уже места в Y, куда X органично вписывается?
   - Есть ли существующий список/форма/секция для подобных элементов?
   - Как выглядят аналогичные элементы в этом же UI?

**Выход:** описание текущего UI-паттерна + где именно должно появиться изменение.

### Шаг 3: Генерация вариантов

На основе шагов 1–2 предложить **2–3 варианта реализации**:

```markdown
### Вариант A: {название}
- **Суть:** ...
- **Плюсы:** ...
- **Минусы:** ...
- **Консистентность с существующим UI:** высокая / средняя / низкая

### Вариант B: {название}
- **Суть:** ...
- **Плюсы:** ...
- **Минусы:** ...
- **Консистентность с существующим UI:** высокая / средняя / низкая

### Рекомендация
Наилучший вариант — **Вариант X**, потому что [обоснование с привязкой к UI-паттерну].
```

**Обязательные критерии сравнения:**
- Консистентность с существующим UI (главный критерий)
- Объём изменений (меньше = лучше)
- Влияние на тесты
- Масштабируемость

### Шаг 4: СТОП-ТОЧКА

**⛔ НЕ НАЧИНАТЬ РЕАЛИЗАЦИЮ** до одобрения пользователя.

Показать пользователю:
1. Карту затронутых файлов (кратко)
2. Текущий UI-паттерн (что есть сейчас)
3. Варианты решения с рекомендацией
4. Ожидать явного одобрения

## Пример применения (кейс «ENV в админы»)

**Задача:** «Добавь ENV-токен в функционал "В админы"»

**Шаг 1 — Карта:** router `management.py`, service `admin_tools_promote.py`, schema `api_payloads.py`, API `management.api.ts`, hook `usePromoteAdminsLogic.ts`, UI `PromoteAdminsSelection.tsx`, modal `PromoteAdminsModal.tsx`.

**Шаг 2 — UI Discovery:**  
Читаю `PromoteAdminsSelection.tsx` → вижу:
- Правая колонка — «Системные страницы» — список `SystemAccount[]` с чекбоксами
- Аккаунты загружаются через `getAllSystemAccounts()` и отображаются как элементы списка
- **Паттерн:** каждый аккаунт — строка с аватаром, именем, ID и чекбоксом

**Шаг 3 — Варианты:**  
- **Вариант A (чекбокс в футере):** Отдельный чекбокс `+ ENV токен` → ❌ нарушает паттерн, ENV не видна в списке  
- **Вариант B (элемент списка):** Загрузить ENV через `verifyEnvToken()`, создать виртуальный `SystemAccount`, вставить в начало списка → ✅ консистентно с UI  
- **Рекомендация: Вариант B** — органично вписывается в существующий список

**Шаг 4:** Показать пользователю → получить одобрение → реализовать.
