# Скилл: Тестировщик (Tester)

## Описание

Комплексное тестирование бэкенда и фронтенда. Проверяет импорты, эндпоинты API, бизнес-логику, типы. Работает после рефакторинга, при добавлении нового функционала, или по конкретной области/папке/модулю по запросу пользователя.

- «протестируй», «покрой тестами», «проверь что работает»
- «smoke-тест», «тест API», «тест эндпоинтов»
- «проверь импорты», «проверь что не сломалось»
- «тест после рефакторинга», «всё ли работает»
- «посмотри логи», «прочитай логи», «что в логах»
- «тест бизнес-логики», «интеграционные тесты»

## Архитектура тестирования

```
tests/
  conftest.py                  ← Фикстуры: TestClient, тестовая БД, авторизация
  smoke/
    test_endpoints.py          ← Smoke-тест всех эндпоинтов (живы ли?)
  imports/
    test_backend_imports.py    ← Проверка импортов Python-модулей
  integration/
    test_<module>.py           ← Интеграционные тесты по модулям
  utils/
    endpoint_collector.py      ← Автосбор эндпоинтов из роутеров
    log_reader.py              ← Чтение и анализ логов
```

> **Тесты НЕ трогают прод и предпрод.** Только локальная БД (SQLite in-memory или файл).

## Процедура 2: Smoke-тест (быстрая проверка после рефакторинга)

**Триггер:** «проверь что не сломалось», «smoke-тест», «тест после рефакторинга»

**Время:** ~10 секунд

### Что проверяет

1. **Импорты** — все .py файлы в целевой папке импортируются без ошибок
2. **Эндпоинты** — все роуты отвечают (не 500)
3. **TypeScript** — `tsc --noEmit` для фронтенда (если затронут)

### Алгоритм

```
1. Определить область (папка / модуль / весь проект)
2. Запустить проверку импортов → scripts/check_imports.py <путь>
3. Запустить smoke-тест эндпоинтов → scripts/smoke_test.py
     (убедиться что uvicorn запущен на localhost:8000)
4. Если затронут фронтенд → запустить tsc --noEmit
5. Собрать результаты → вывести отчёт
```

### Формат отчёта

```
═══════════════════════════════════════
  SMOKE-ТЕСТ: <область>
═══════════════════════════════════════

📦 ИМПОРТЫ (Python)
  ✅ services.automations.stories.viewers_details
  ✅ services.automations.stories.publish
  ❌ services.automations.stories.analytics — ImportError: cannot import name 'X'

🌐 ЭНДПОИНТЫ (28 роутеров)
  ✅ GET  /api/projects/           200
  ✅ GET  /api/posts/              200  (с ?project_id=1)
  ❌ POST /api/automations/start   500  ← СЛОМАН

🔤 TYPESCRIPT
  ✅ tsc --noEmit: 0 ошибок
  — или —
  ❌ tsc --noEmit: 3 ошибки (список)

═══════════════════════════════════════
  ВЕРДИКТ: ❌ 2 проблемы найдены
═══════════════════════════════════════
  1. ImportError в analytics — вероятно переименован экспорт
  2. 500 на /automations/start — нужно проверить логику
```

## Процедура 3: Интеграционные тесты (глубокая проверка)

**Триггер:** «покрой тестами», «тест бизнес-логики», «интеграционные тесты»

**Время:** 1-5 минут (зависит от объёма)

### Алгоритм

```
1. Определить область тестирования
2. Прочитать код целевого модуля (роутер + сервис + crud + схемы)
3. Составить список тест-кейсов:
   a. CRUD-операции (создать → прочитать → обновить → удалить)
   b. Валидация (невалидные данные → ожидаем 422)
   c. Граничные случаи (пустые данные, несуществующие ID)
   d. Авторизация (без токена → 401, чужие данные → 403)
4. Написать тесты в tests/integration/test_<модуль>.py
5. Запустить: pytest tests/integration/test_<модуль>.py -v
6. Прочитать вывод, составить вердикт
```

## Процедура 5: Чтение и анализ логов

**Триггер:** «посмотри логи», «что в логах», «прочитай логи»

### Локальные логи (бэкенд)

```
1. Прочитать вывод терминала uvicorn (get_terminal_output)
2. Найти ERROR, WARNING, Traceback
3. Если есть файл логов → прочитать последние 100 строк
```

### Логи предпрода

```
1. Прочитать shared/config.ts → взять URL предпрода
2. Запросить эндпоинт здоровья: GET /api/health (если есть)
3. Запросить проблемный эндпоинт напрямую через curl/httpx
4. Проанализировать ответ (статус, тело, заголовки)
```

### Фронтенд-логи

```
1. Проверить tsc --noEmit — ошибки типов
2. Проверить npm run build — ошибки сборки
3. Если пользователь описал ошибку в консоли браузера → проанализировать
```

## Порядок приоритетов при тестировании

```
1. 🔴 Импорты — быстро, ловит 60% проблем рефакторинга
2. 🔴 Эндпоинты (smoke) — ловит сломанные роуты и схемы
3. 🟡 TypeScript (tsc) — ловит проблемы типов на фронте
4. 🟡 Сборка (build) — финальная проверка фронта
5. 🟢 Интеграционные тесты — глубокая проверка логики
6. 🟢 Логи — диагностика конкретных проблем
```

---

## Правила

1. **Тесты НЕ трогают прод/предпрод БД** — только локальная SQLite (in-memory или файл)
2. **Не устанавливать vitest без явного запроса** — `tsc --noEmit` достаточно для 90% случаев
3. **Всегда показывать вердикт** — не просто «тесты прошли», а конкретно что проверено и что в порядке
4. **Smoke перед интеграционными** — сначала быстрая проверка, потом глубокая
5. **Убедиться что uvicorn запущен** перед тестом эндпоинтов
6. **Логировать проблемы чётко** — файл, строка, что сломалось, почему, как починить

## Делегирование саб-агентам

Следующие шаги рекомендуется выполнять через `runSubagent` для экономии контекста:

| Шаг | Что делегировать | Промпт для саб-агента |
|---|---|---|
| ⚡ Smoke — Python-импорты | Проверка всех импортов Python-модулей | «Запусти scripts/check_imports.py или прочитай backend_python/ и проверь все import-ы. Верни список: файл, строка, ошибка импорта» |
| ⚡ Smoke — TypeScript | Проверка компиляции фронтенда | «Запусти tsc --noEmit или используй get_errors для всех .tsx/.ts файлов. Верни список ошибок: файл, строка, сообщение» |
| Анализ бэкенд-стека | Чтение роутер → сервис → crud → модель | «Прочитай бэкенд-стек для {модуль}: routers/{модуль}.py, services/{модуль}.py, crud/{модуль}.py, schemas/{модуль}.py. Верни: эндпоинты, типы данных, SQL-запросы, потенциальные N+1» |
| Анализ фронтенд-стека | Чтение компонент → хук → API → типы | «Прочитай фронтенд-стек для {модуль}: features/{модуль}/. Верни: компоненты, хуки, API-вызовы, типы, обработку ошибок» |

**⚡ ПАРАЛЛЕЛЬНО:**
- «Python-импорты» + «TypeScript» — запускать одновременно (разные языки, разные проверки)
- «Анализ бэкенд-стека» + «Анализ фронтенд-стека» — запускать одновременно (разные части кодовой базы)
