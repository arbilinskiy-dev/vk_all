# UI/UX Рефакторинг — Параллельная архитектура

Скилл выполняет аудит области интерфейса через **массовую параллелизацию**: каждая проверка — отдельный саб-агент.

## Архитектура: 26 параллельных проверок

Дизайн-система содержит **13 общих концепций** + **13 частных эталонов** = **26 проверок**.

## Процедура аудита

### Фаза 1: Разведка (1 саб-агент)

Определить область проверки (спросить пользователя, если не указано). Запустить саб-агента:

**Промпт саб-агенту «Разведчик»:**
> Прочитай ВСЕ файлы в {область}: .tsx компоненты, хуки, стили.
> Для каждого файла верни:
> 1. Полный путь
> 2. Количество строк
> 3. Какие типы UI-элементов есть: кнопки, табы, скроллбары, тумблеры, счётчики, модалки, SVG-иконки, изображения, редактор, поиск, дропдауны, info-панель
> 4. Есть ли: хардкод цветов (hex/rgb), inline-стили, alert/confirm/prompt, числа без склонения
> 5. Есть ли инлайн-алерты: паттерн `useState` + `setTimeout` + показ/скрытие `<div className="bg-green-50">` / `bg-red-50` / `bg-yellow-50` / `bg-blue-50` для временных уведомлений (должны быть заменены на toast)
> 6. Какие shared-компоненты импортируются
> Формат: JSON-массив файлов

**Результат:** карта области — что есть, что проверять.

### Фаза 2: Формирование чеклиста

На основе карты из Фазы 1 определить, какие из проверок РЕЛЕВАНТНЫ данной области:
- Если в области нет тумблеров — проверка E3 пропускается
- Если нет текстового редактора — E9 и E10 пропускаются
- **Общие концепции (C1–C13) проверяются ВСЕГДА**
- **Частные эталоны (E1–E13) — только если в области есть соответствующий элемент**

Итог: список из N проверок (13 ≤ N ≤ 26).

### Фаза 3: Параллельный аудит (N саб-агентов одновременно)

Для каждой релевантной проверки запустить ОТДЕЛЬНОГО саб-агента. Все саб-агенты запускаются **параллельно**.

**Шаблон промпта для каждого саб-агента:**

> Ты — аудитор UI/UX. Проверяешь ОДНО правило дизайн-системы.
>
> **Правило:** {ID} — {название}
> **Файл эталона:** Прочитай `c:\Users\nikita79882\Desktop\vk planer code\12.02.2026\.agents\skills\uiux-refactor\references\{файл_эталона}`
> **Файлы для проверки:** {список файлов из карты Фазы 1, релевантных этой проверке}
>
> Прочитай файл эталона ПОЛНОСТЬЮ. Затем прочитай каждый файл для проверки.
> Сопоставь код с эталоном.
>
> Верни ответ СТРОГО в формате:
> ```
> ПРАВИЛО: {ID} — {название}
> СТАТУС: ✅ OK | ❌ НАРУШЕНИЕ
> НАРУШЕНИЯ (если есть):
> - Файл: {путь}, строка: {N}
>   Текущий код: {что сейчас}
>   Эталон: {что должно быть}
>   Исправление: {конкретное действие}
> ```
> Если нарушений нет — верни только СТАТУС: ✅ OK.
> НЕ трогай бизнес-логику. Проверяй ТОЛЬКО визуальное соответствие.

**Маппинг проверка → файлы:**
- **C1 (чеклист):** все файлы области — общий проход
- **C2 (цвета):** все файлы — grep на hex/rgb/#, hsl
- **C3 (приоритет компонентов):** все файлы — анализ импортов
- **C4 (запреты):** все файлы — grep на alert, confirm, prompt, !important, style=
- **C5 (анимации):** все файлы с conditional rendering (&&, ternary, show/hide), все файлы с числовыми показателями (проверить AnimatedNumber), все SVG-графики (проверить draw/fade анимации), все isLoading (проверить отсутствие overlay/opacity)
- **C6 (склонение):** все файлы с числами/счётчиками в JSX
- **C7 (сайдбар):** файлы сайдбара/навигации (если область затрагивает)
- **C8 (Tailwind spacing):** все файлы — grep на `w-4.5|h-4.5|w-5.5|h-5.5|p-4.5|m-4.5|gap-4.5`
- **C9 (overflow+absolute):** все файлы с `absolute` + отрицательными offsets — проверить overflow предков
- **C10 (полнота фикса):** ПРАВИЛО ПРОЦЕССА: после каждого фикса grep по файлу на аналогичный паттерн
- **C11 (blur race):** все `onBlur={handle...}` — проверить relatedTarget guard
- **C12 (минимальный diff):** ПРАВИЛО ПРОЦЕССА: не менять layout-модель для точечного бага
- **E6 (уведомления):** ВСЕ файлы области — grep на паттерны инлайн-алертов:
  - `bg-green-50 border.*green-200` + `useState.*show` + `setTimeout` — транзиентный инлайн-алерт, заменить на `useToast().success()`
  - `bg-red-50 border.*red-200` + `useState.*show` + `setTimeout` — заменить на `useToast().error()`
  - `bg-yellow-50 border.*yellow-200` + аналогичный паттерн — заменить на `useToast().warning()`
  - Любой `<div>` с conditional rendering (`{showX && <div className="...bg-...-50 border...">`) + `setTimeout(() => setShowX(false), ...)` — это инлайн-алерт, заменить на toast
  - **Исправление:** убрать `useState` для видимости, убрать `setTimeout`, убрать JSX-блок, добавить `import { useToast } from '...ToastProvider'`, вызывать `toast.success()` / `toast.error()` / `toast.warning()` в обработчике
  - **Не путать** с постоянными информационными панелями (без `setTimeout`, без toggle-стейта) — их НЕ трогать
- **E1–E5, E7–E13:** только файлы, содержащие соответствующий UI-элемент (из карты Фазы 1)

> SVG-performance (C13–C15) вынесены в скилл `arch-review` (пункт #11).

Собрать все ответы саб-агентов в единую таблицу:

```
| # | Правило | Статус | Нарушений | Краткое описание |
|---|---------|--------|-----------|-----------------|

```

Под таблицей — детализация каждого нарушения с конкретными строками и исправлениями.

### Фаза 5: Параллельное исправление

Сгруппировать все нарушения **по файлам** (не по правилам). Для каждого файла с нарушениями — один саб-агент:

**Промпт для саб-агента-фиксера:**
> Исправь нарушения дизайн-системы в файле {путь}.
> Список нарушений:
> {все нарушения для этого файла из отчёта}
> Для каждого исправления прочитай эталон из references/{файл}.
> НЕ трогай бизнес-логику. Только визуальное соответствие.
> Комментарии в коде — на русском языке.

Если нарушения в разных файлах — саб-агенты-фиксеры запускаются **параллельно**.
Если все нарушения в одном файле — один саб-агент исправляет всё.

### Фаза 6: Финальная проверка (1 саб-агент)

После всех исправлений — запустить саб-агента для повторного прохода:

> Прочитай файлы {список исправленных файлов}.
> Проверь по чеклисту нарушений из предыдущего аудита: {список нарушений}.
> Для каждого нарушения проверь: исправлено ли? Не появились ли новые?
> Верни: ✅ Все исправлено / ❌ Осталось: {что}

## Пост-фикс валидация (ОБЯЗАТЕЛЬНО после каждого исправления)

После ЛЮБОГО изменения CSS/Tailwind классов — выполнить валидацию:

### 1. Grep на невалидные Tailwind-классы
```
# Несуществующие half-step классы (ЗАПРЕЩЕНЫ)
w-4.5|h-4.5|w-5.5|h-5.5|p-4.5|m-4.5|gap-4.5|space-x-4.5|space-y-4.5
```
Допустимые half-step: ТОЛЬКО `0.5, 1.5, 2.5, 3.5`. После `3.5` → целые: `4, 5, 6, 7, 8...`

### 2. Grep на аналогичные баги
Если исправил `w-4.5`→`w-5` в строке 120, выполни:
```
grep 'w-4\.5' в ЭТОМ ЖЕ файле
```
Исправь ВСЕ вхождения, не только упомянутое пользователем.

### 3. Проверка overflow для absolute-элементов
Если добавил элемент с `absolute` + отрицательные offsets (`-top-*`, `-right-*`):
- Найди ближайший родительский `overflow-*` (auto/hidden/scroll)
- Если есть — элемент будет ОБРЕЗАН. Исправь overflow или измени позиционирование.

### 4. Проверка blur на inline-формах
Если добавил `onBlur={handleSave}` на input рядом с кнопками:
- Добавь guard: `if (container.contains(e.relatedTarget)) return;`
- Или `onMouseDown={e => e.preventDefault()}` на соседних кнопках

## Важно

- НЕ трогать бизнес-логику — только визуальное соответствие дизайн-системе
- НЕ менять структуру компонента, если это не требуется правилами
- Если в `ui-kit/` или `shared/components/` есть готовый компонент — предложить замену
- Комментарии в коде — на русском языке
- Количество проверок = 12 концепций + 13 эталонов = **25 максимум**. SVG-performance проверяется в `arch-review`. При добавлении новых reference-файлов — обновить таблицы выше

## Резюме параллелизации

| Фаза | Саб-агентов | Параллельно? | Что делает |
|------|------------|-------------|------------|
| 1. Разведка | 1 | — | Сканирует область, строит карту UI-элементов |
| 2. Чеклист | 0 (основной агент) | — | Фильтрует 25 проверок → N релевантных |
| 3. Аудит | N (12–25) | ✅ ВСЕ параллельно | Каждый проверяет одно правило |
| 4. Отчёт | 0 (основной агент) | — | Сводит результаты в таблицу |
| 5. Исправление | M (по числу файлов) | ✅ ВСЕ параллельно | Каждый правит один файл |
| 6. Финалка | 1 | — | Верификация исправлений |

**Итого:** вместо последовательного аудита (25 пунктов × 2+ минуты = 50+ минут) — параллельный (1 + 1 + N×1 + 1 = ~4 шага).
