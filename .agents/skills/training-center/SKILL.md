# Training Center — Разработка Центра обучения

## Область действия

Работай **ТОЛЬКО** с файлами в `features/training/`. Не модифицируй другие части проекта.

## Архитектура

```
features/training/
├── components/
│   ├── TrainingPage.tsx      ← Главная страница (НЕ ТРОГАТЬ)
│   ├── TableOfContents.tsx   ← Оглавление (НЕ ТРОГАТЬ)
│   ├── TopicContent.tsx      ← Роутер контента ← РЕГИСТРАЦИЯ НОВЫХ РАЗДЕЛОВ
│   └── content/              ← РАБОЧАЯ ПАПКА
│       ├── PlaceholderPage.tsx
│       ├── PostCardMocks.tsx     ← Mock-компоненты для постов
│       ├── SidebarMocks.tsx      ← Mock-компоненты для сайдбара
│       ├── PostCardDeepDive.tsx  ← Пример готового раздела
│       └── SidebarNavDeepDive.tsx
└── data/
    └── tocData.ts            ← Структура оглавления
```

## Процедура создания нового раздела (5 шагов)

### Шаг 1 — Изучить код

1. Пользователь указывает раздел («разберём Предложенные посты»)
2. **Найти** все файлы функционала (компоненты, хуки, сервисы)
3. **Проанализировать**: состояния, обработчики, props, зависимости
4. **Выявить** неочевидные механики (drag-and-drop, двойной клик, скрытые кнопки)

> **КРИТИЧЕСКОЕ ПРАВИЛО:** Описывать ТОЛЬКО то, что реально найдено в коде. Не фантазировать.

### Шаг 2 — Объяснить разработчику

Подробно объяснить:
- Как устроен функционал
- Какие есть состояния и переходы
- Неочевидные особенности
- Что важно показать пользователям

### Шаг 3 — Создать контент

1. Определить `path` из `features/training/data/tocData.ts`
2. Создать компонент в `features/training/components/content/`
3. Создать mock-компоненты в отдельном файле `*Mocks.tsx`

Прочитать:
- [references/writing_style.md](references/writing_style.md) — стиль текста и замена тех. терминов
- [references/component_template.md](references/component_template.md) — шаблон + стилизация + Sandbox
- [references/mock_components.md](references/mock_components.md) — создание mock-компонентов

### Шаг 4 — Зарегистрировать

1. Добавить импорт в `features/training/components/TopicContent.tsx`
2. Добавить в `componentMap`, сопоставив с `path` из tocData

```tsx
import { НовыйРаздел } from './content/НовыйРаздел';

const componentMap = {
    // ...существующие
    'путь-из-tocData': НовыйРаздел,
};
```

### Шаг 5 — Чеклист

- [ ] Компонент в `features/training/components/content/`
- [ ] Использует `prose prose-indigo max-w-none`
- [ ] Интерактивные блоки обёрнуты в `not-prose`
- [ ] Песочницы через компонент `Sandbox`
- [ ] Mock'и в отдельном файле `*Mocks.tsx`
- [ ] Зарегистрирован в `TopicContent.tsx`
- [ ] Path совпадает с `tocData.ts`

## Запреты

1. НЕ модифицировать файлы вне `features/training/`
2. НЕ импортировать реальные компоненты — только mock-версии
3. НЕ подключаться к API или контекстам приложения
4. НЕ использовать `window.confirm()` или нативные диалоги
5. НЕ придумывать функционал, которого нет в коде

## Делегирование саб-агентам

Для экономии контекстного окна следующие шаги рекомендуется выполнять через `runSubagent`:

| Шаг | Что делегировать | Промпт для саб-агента |
|---|---|---|
| ⚡ Шаг 1 — Изучить код функционала | Поиск и анализ всех файлов функционала (10–30 файлов) | «Найди все файлы в features/{название}/. Для каждого выпиши: экспортируемые компоненты, props, useState/useEffect, обработчики событий, неочевидные механики. Верни структурированный отчёт» |
| ⚡ Шаг 3 — Чтение справочников | Загрузка writing_style.md, component_template.md, mock_components.md | «Прочитай 3 файла из references/. Верни конспект: ключевые правила, шаблон кода, запреты, примеры замены терминов» |
| Шаг 3 — Генерация mock-компонентов | Создание файла *Mocks.tsx (>200 строк) | «Сгенерируй файл {Name}Mocks.tsx. Нужны mock-версии: [список]. Каждый mock: автономный, без API/контекстов, с useState, стили Tailwind, обёртка not-prose. Верни полный файл» |
| Шаг 5 — Валидация | Перекрёстная проверка 3–4 файлов | «Проверь раздел {название}: 1) .tsx использует prose prose-indigo 2) not-prose на интерактивных блоках 3) Sandbox компонент 4) Моки в *Mocks.tsx 5) componentMap в TopicContent.tsx 6) path совпадает с tocData.ts. Верни ✅/❌ по каждому пункту» |

**⚡ ПАРАЛЛЕЛЬНО:** «Изучить код функционала» (Шаг 1) + «Чтение справочников» (Шаг 3) — запускать **одновременно** (один анализирует реальный код в features/, другой читает reference-файлы). «Генерация mock-компонентов» — после Шага 1 (нужно знать какие компоненты мокать).
