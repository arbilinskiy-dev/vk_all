# Bug Killer — Системная диагностика и устранение багов

## Философия

Каждый баг — это **расследование**. Не гадать, а собирать улики, строить гипотезы, проверять их и только потом чинить. Если фикс не помог — копать глубже, пока проблема не будет устранена полностью.

---

## Фаза 0: Приём входных данных

Пользователь может предоставить любую комбинацию:

| Тип входных данных | Что с ним делать |
|---|---|
| Текстовое описание | Извлечь: что ожидалось → что происходит → когда началось |
| Логи терминала сервера | Найти ERROR, Traceback, WARNING, статус-коды |
| Логи консоли браузера | Найти ошибки JS, CORS, Network, React errors |
| Скриншот поведения | Проанализировать визуальные аномалии, сопоставить с UI-компонентами |
| Код или файл | Проанализировать на антипаттерны, ошибки логики |

### Если данных недостаточно — запросить:

```
Для точной диагностики нужна дополнительная информация:
1. [конкретный вопрос]

```

Максимум 3 уточняющих вопроса за раз. Если можно начать расследование без ответов — начинать.

## Фаза 2: Расследование (сбор улик)

### Стратегия расследования по домену

#### Фронтенд

```
1. Найти компонент по описанию:
   - grep_search по тексту ошибки, имени компонента, роуту
   - Прочитать компонент и его зависимости (хуки, контексты, утилиты)

2. Проследить поток данных:
   - Откуда данные? (API-вызов → хук → компонент → рендер)
   - Где обрыв? (запрос не уходит / ответ неверный / стейт не обновляется / рендер ломается)

3. Проверить типы:
   - Совпадают ли типы фронта с ответом API?
   - Есть ли optional-поля, которые приходят как null/undefined?
```

#### Бэкенд

```
1. Найти эндпоинт по ошибке:
   - URL запроса → найти роутер в routers/
   - Проследить: роутер → сервис → CRUD → модель → БД

2. Проверить каскад:
   - Валидация входных данных (Pydantic-схемы)
   - Бизнес-логика в сервисе
   - SQL-запрос в CRUD
   - Модель и миграции

3. Проверить edge-cases:
   - Что если данных нет? (None, пустой список)
   - Что если данные невалидны?
   - Что если параллельные запросы?
```

#### API-взаимодействие (фронт↔бэк)

```
1. Сравнить контракт:
   - Что отправляет фронт (тело запроса, заголовки)?
   - Что ожидает бэк (Pydantic-схема)?
   - Что возвращает бэк (ответ)?
   - Что ожидает фронт (TypeScript-тип)?

2. Проверить конфигурацию:
   - Правильный ли BASE_URL в shared/config.ts?
   - CORS настроен на бэке?
   - Авторизация (JWT) передаётся?

3. Проверить прод через MCP (быстрая диагностика):
   - mcp_fetch GET/POST к api.dosmmit.ru — проверить ответ
   - Сравнить ответ прода с ожидаемой Pydantic-схемой
   - Если ответ 4xx/5xx — сразу видна проблема без SSH
```

#### БД / данные

```
1. Проверить модель (models.py, models_library/)
2. Проверить миграции (db_migrations/)
3. Проверить состояние БД:
   - Локально: SQLite MCP (если подключен) или Python-запрос
   - Прод: PostgreSQL MCP через SSH-туннель (если подключен)
   - Fallback: SSH → docker exec → SQL-запрос
```

### Делегирование саб-агентам

Следующие шаги выполнять через `runSubagent` для экономии контекста:

| Шаг | Промпт для саб-агента |
|---|---|
| Массовый поиск по кодовой базе | «Найди все файлы, которые импортируют/используют X. Верни список: путь, строка, контекст (3 строки вокруг). Формат — таблица markdown.» |
| ⚡ Анализ каскада бэкенда | «Прочитай файлы [роутер, сервис, crud, схема, модель] и опиши полный поток данных для эндпоинта X. Верни: входные данные → валидация → бизнес-логика → SQL → ответ.» |
| ⚡ Анализ потока фронтенда | «Прочитай [компонент, хук, API-сервис, типы] и опиши поток данных: API-вызов → стейт → рендер. Верни блок-схему в текстовом виде.» |
| Проверка типов фронт↔бэк | «Сравни Pydantic-схему [файл] и TypeScript-тип [файл]. Найди расхождения. Верни таблицу: поле, тип в Python, тип в TS, совпадает ли.» |

**⚡ ПАРАЛЛЕЛЬНО:** «Анализ каскада бэкенда» + «Анализ потока фронтенда» — запускать одновременно (независимые части кодовой базы). «Проверка типов фронт↔бэк» — после получения результатов обоих.

## Фаза 3: Гипотезы и дебаг-логи

### Шаг 3.1: Сформулировать гипотезы

**💡 Для сложных багов используй Sequential Thinking (MCP):**

Если баг затрагивает ≥2 домена (фронт + бэк, или бэк + VK API), или если после первого расследования причина неочевидна — используй `sequential-thinking` для пошагового анализа:

```
→ sequential-thinking: thought
  "Собранные улики:
   1. Фронт получает 200, но данные пустые
   2. Бэк-лог показывает: query вернул 15 записей
   3. Pydantic-схема Response содержит поле items: list[PostResponse]
   
   Гипотеза A: Сериализация — поле в модели не совпадает со схемой
   Гипотеза B: Фильтрация на бэке — данные есть, но обрезаются условием
   Гипотеза C: Фронт парсит ответ неправильно (ожидает .data.items, получает .items)
   
   Проверка A: сравнить model.Post атрибуты vs PostResponse поля
   Проверка B: добавить лог после фильтрации
   Проверка C: console.log(response) на фронте
   
   Наиболее вероятна A — начинаю с неё."
```

Это экономит токены основного чата и структурирует мышление при нескольких гипотезах.

На основе собранных данных сформулировать 1–3 гипотезы:

```
═══════════════════════════════════════
  ГИПОТЕЗЫ
═══════════════════════════════════════

🔍 Гипотеза 1 (вероятность: высокая)
   Причина: [описание]
   Улики: [что указывает на это]
   Проверка: [как подтвердить/опровергнуть]

🔍 Гипотеза 2 (вероятность: средняя)
   Причина: [описание]
   Улики: [что указывает на это]
   Проверка: [как подтвердить/опровергнуть]
```

### Шаг 3.2: Добавить дебаг-логи (если нужно)

Если причина не очевидна — добавить временные дебаг-логи для подтверждения гипотезы.

#### Бэкенд (Python)

```python
import logging
logger = logging.getLogger(__name__)

# Добавить в подозрительное место:
logger.debug(f"[BUG-KILLER DEBUG] <описание>: {переменная=}")
```

#### Фронтенд (React/TS)

```typescript
// Добавить в подозрительное место:
console.log('[BUG-KILLER DEBUG] <описание>:', переменная);
```

**Правила дебаг-логов:**
- Префикс `[BUG-KILLER DEBUG]` — чтобы легко найти и удалить потом
- Минимум логов — только то, что проверяет конкретную гипотезу
- После решения бага — **обязательно удалить** все дебаг-логи

## Фаза 4: Фикс

### Шаг 4.1: Применить исправление

Исправить **первопричину**, а не симптом:

```
❌ Плохо: добавить try/except чтобы скрыть ошибку
✅ Хорошо: исправить логику, которая вызывает ошибку

❌ Плохо: добавить || '' чтобы не крашилось на undefined
✅ Хорошо: разобраться почему приходит undefined и починить источник
```

### Шаг 4.2: Проверить побочные эффекты

```
1. Какие другие компоненты/модули используют изменённый код?
   → grep_search по имени функции/компонента
2. Не сломает ли фикс другой функционал?
   → Проследить зависимости
3. Нужна ли миграция БД?
   → Если изменилась модель — да
```

### Шаг 4.3: Удалить дебаг-логи

Если на Шаге 3.2 были добавлены дебаг-логи — удалить **все** строки с `[BUG-KILLER DEBUG]`.

## Правила

1. **Не гадать — расследовать.** Всегда собирать данные перед фиксом.
2. **Чинить причину, не симптом.** Разобраться в корне проблемы.
3. **Дебаг-логи временные.** Добавлять с префиксом `[BUG-KILLER DEBUG]`, удалять после решения.
4. **Минимальный фикс.** Не рефакторить полпроекта ради одного бага.
5. **Проверять побочные эффекты.** grep по изменённым функциям/компонентам.
6. **Итерировать до победы.** Если фикс не помог — копать глубже, не сдаваться.
7. **Не трогать прод-данные.** Только чтение логов и состояния. Запись — только через деплой.
8. **Всегда показывать отчёт.** Пользователь должен понимать что найдено, что починено и как проверить.
