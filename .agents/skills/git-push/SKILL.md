# Скилл: Пуш на GitHub (git-push)

## Описание

Анализ изменений в проекте, формирование осмысленного коммита и пуш на GitHub. AI смотрит текущее состояние репозитория, сравнивает с тем что на GitHub, группирует и описывает изменения, делает коммит и пушит.

## Ключевые слова активации

- «пуш», «пуш на гитхаб», «push», «закинь на гитхаб»
- «закоммить», «сделай коммит», «commit»
- «сохрани на гитхаб», «залей изменения»
- «что изменилось» (+ контекст git)

## Конфигурация репозитория

```
Remote:  origin → https://github.com/arbilinskiy-dev/vk_all.git
Branch:  main
```

## Процедура 1: Полный пуш (основная)

**Триггер:** «пуш на гитхаб», «закинь на гитхаб», «push»

### Шаг 1: Анализ состояния

```bash
# Проверяем что на remote
git fetch origin

# Смотрим есть ли расхождения с remote
git log origin/main..HEAD --oneline

# Смотрим все локальные изменения
git status --short
```

**Если есть незапушенные коммиты** → сначала показать их пользователю.
**Если есть конфликты с remote** → предупредить, предложить pull.

### Шаг 2: Анализ изменений

```bash
# Группируем изменения по областям
git status --short
```

Категоризировать файлы:

| Префикс | Область |
|---------|---------|
| `backend_python/` | Бэкенд |
| `features/` | Фронтенд (фичи) |
| `shared/` | Фронтенд (общее) |
| `services/` | Фронтенд (сервисы) |
| `hooks/` | Фронтенд (хуки) |
| `docs/` | Документация |
| `.agents/` | Скиллы/инструкции AI |
| `.github/` | Конфигурация |
| Остальное | Корень проекта |

### Шаг 3: Формирование описания коммита

**Формат:**

```
<тип>: <краткое описание на русском>

Изменения:
- <область 1>: что изменилось (N файлов)
- <область 2>: что изменилось (N файлов)
...

Файлов изменено: X | Новых: Y | Удалено: Z
```

**Типы коммитов:**

| Тип | Когда |
|-----|-------|
| `feat` | Новый функционал |
| `fix` | Исправление бага |
| `refactor` | Рефакторинг без изменения логики |
| `docs` | Только документация |
| `style` | Стили, UI без логики |
| `chore` | Конфигурация, зависимости, скрипты |
| `mix` | Несколько типов изменений |

### Шаг 4: Показать отчёт пользователю

```
═══════════════════════════════════════
  КОММИТ И ПУШ НА GITHUB
═══════════════════════════════════════

📦 Репозиторий: arbilinskiy-dev/vk_all
🌿 Ветка: main

📊 ИЗМЕНЕНИЯ:
  Бэкенд:     45 файлов (CRUD, сервисы, роутеры)
  Фронтенд:   32 файла (посты, автоматизации)
  Документация: 8 файлов
  Скиллы AI:   12 файлов (новые скиллы)
  ─────────────
  Итого:       97 файлов

📝 КОММИТ:
  mix: Рефакторинг CRUD, новые скиллы AI, обновление документации

═══════════════════════════════════════
  Выполнить? (да/нет)
═══════════════════════════════════════
```

### Шаг 5: Выполнение (только после одобрения!)

```bash
# Добавляем всё
git add -A

# Коммитим с описанием
git commit -m "<сообщение>"

# Пушим
git push origin main
```

### Шаг 6: Подтверждение

```bash
# Проверяем что пуш прошёл
git log origin/main --oneline -1
```

Сообщить результат пользователю.

## Процедура 2: Выборочный коммит

**Триггер:** «закоммить только бэкенд», «запуш только фронтенд», «пуш конкретные файлы»

### Алгоритм

```
1. Определить область (по запросу пользователя)
2. git add <конкретные файлы или папки>
3. Сформировать описание (только для выбранных файлов)
4. Показать что будет закоммичено → ждать одобрения
5. git commit + git push
```

---

## Процедура 3: Просмотр изменений (без пуша)

**Триггер:** «что изменилось», «покажи diff», «что не запушено»

### Алгоритм

```
1. git fetch origin
2. git status --short → сгруппировать по областям
3. git diff --stat → сколько строк изменено
4. git log origin/main..HEAD --oneline → незапушенные коммиты
5. Вывести отчёт (без выполнения коммита)
```

---

## Процедура 4: Синхронизация с remote

**Триггер:** «подтяни с гитхаба», «pull», «синхронизируй»

### Алгоритм

```
1. git fetch origin
2. git log HEAD..origin/main --oneline → что нового на remote
3. Если есть изменения:
   a. Есть ли локальные незакоммиченные? → предупредить
   b. git pull origin main --rebase (или --merge)
4. Если конфликты → показать список, помочь решить
```

## Правила

1. **ВСЕГДА показывать отчёт перед коммитом** — не коммитить молча
2. **ВСЕГДА ждать одобрения** — не пушить без «да» от пользователя
3. **Не пушить секреты** — проверять что .env, токены, ключи не попадают в коммит (см. .gitignore)
4. **Осмысленные сообщения** — не "update", не "fix", а конкретное описание
5. **Один коммит = одна логическая единица** — если изменения разнородные, предложить разбить
6. **Проверить .gitignore** — перед первым пушем убедиться что чувствительные файлы игнорируются
7. **git fetch перед пушем** — проверить нет ли конфликтов с remote

## Чувствительные файлы (НЕ должны попасть в коммит)

```
.env
*.key
*_secret*
backend_python/config.py     ← содержит SECRET_KEY и токены
vk_api_methods_tokens.json   ← токены VK
logs_sync.json               ← логи
```

> **Перед каждым пушем** пробегись по `git status` и убедись что эти файлы в .gitignore или не в staging.
