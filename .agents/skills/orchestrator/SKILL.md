# Orchestrator — Оркестрация сложных задач

## Когда активировать

- Задача затрагивает **≥3 модуля** (бэкенд + фронтенд + инфраструктура)
- Задача затрагивает **≥2 модуля** И включает **изменение UI-компонента** (добавление элемента в существующий интерфейс)
- Задача добавляет **новую сущность в БД** + API + фронтенд
- Нужен **полный цикл**: анализ → проектирование → реализация → тестирование
- Пользователь описывает **бизнес-задачу**, а не конкретный файл
- Явные триггеры: «большая задача», «новая фича», «полный цикл»

### Лёгкий режим (2 модуля)

Если задача затрагивает ровно 2 модуля — запускать **сокращённый цикл**:
- Аналитик → Архитектор → один разработчик (backend-dev или frontend-dev) → QA
- Пропускаются: DevOps, UI/UX инженер, Документатор
- Это предотвращает промахи на задачах средней сложности без overhead полного оркестратора

## 🔴 КРИТИЧЕСКИЕ ПРАВИЛА БЕЗОПАСНОСТИ

Проект **VK Content Planner** — production-система, которой пользуются реальные люди. Любое изменение должно быть безопасным.

### Что НЕЛЬЗЯ ломать (Red Zone)

| # | Компонент | Почему критичен |
|---|---|---|
| 1 | `EncryptedString` (models_library/types.py) | Шифрует все VK-токены. Сломать = потеря доступа ко всем сообществам |
| 2 | `ENCRYPTION_KEY` (config.py) | Ключ Fernet. Потеря = невозвратная потеря всех зашифрованных данных |
| 3 | `session_token` генерация (secrets.token_urlsafe) | Замена на слабый генератор = предсказуемые сессии |
| 4 | `auth_middleware.py` — проверка is_active + таймаут | Убрать проверку = протухшие сессии живут вечно |
| 5 | `PERMANENT_ERROR_CODES` в api_client.py | Удалить код 5 = бесконечный retry с невалидным токеном → бан VK |
| 6 | `read=0` в Retry (api_client.py) | Изменить на read>0 = дубликаты постов при сетевых проблемах |
| 7 | `SessionLocal.remove()` в фоновых задачах | Убрать = утечка DB-соединений, deadlock |
| 8 | Каскады `ondelete="CASCADE"` на projects | Изменить = orphan-записи, целостность данных нарушена |
| 9 | Redis-лок `vk_callback:worker_lock` | Убрать = двойная обработка событий, дубликаты сообщений |
| 10 | SSE `_instance_id` фильтр | Убрать = двойная доставка событий клиентам |

### Паттерны БД, которые ОБЯЗАТЕЛЬНО соблюдать

- **Сессия передаётся параметром** `db: Session` — не создаётся внутри CRUD
- **Commit в CRUD**, не в роутере — каждая CRUD-функция управляет транзакцией
- **`subqueryload`/`joinedload`** для связей — никогда ленивая загрузка в списках
- **`synchronize_session=False`** в bulk-delete — ORM identity map не обновляется
- **PK — UUID строки** (`str(uuid.uuid4())`), НЕ автоинкремент (кроме TokenLog, VkProfile, AuthLog)
- **Все FK к projects — `ondelete="CASCADE"`**
- **Шифрование через EncryptedString** для всех токенов/ключей

### Паттерны фронтенда, которые ОБЯЗАТЕЛЬНО соблюдать

- **API через `callApi`** (shared/utils/apiClient.ts) — НИКОГДА прямой `fetch`
- **Tailwind CSS** — никакого inline style или CSS-файлов
- **Тёмная тема**: bg-gray-800/900, text-gray-100/300, border-gray-600/700
- **Типизация**: НЕ использовать `as any`, всегда определять интерфейсы
- **Хуки**: логика >30 строк → выносить в `use*.ts`
- **Shared-компоненты**: `ConfirmationModal`, `SlidePanel`, `LazyImage` — переиспользовать

## Архитектура проекта (справка для агентов)

### Бэкенд (FastAPI + SQLAlchemy)

```
backend_python/
├── main.py                    ← Startup: миграции → сидинг → APScheduler → Callback Worker → SSE
├── config.py                  ← ENV: encryption_key, vk_user_token, gemini_api_key, redis_*
├── database.py                ← get_db() (FastAPI dependency) + get_background_session() (threads)
├── models.py + models_library/ ← 54 таблицы, EncryptedString для токенов
├── routers/                   ← ~20 роутеров (тонкие — парсинг → вызов сервиса → ответ)
├── services/                  ← Бизнес-логика, VK API клиент, SSE Manager, Scheduler
│   ├── vk_api/                ← api_client (retry, permanent errors), token_manager (ротация)
│   ├── vk_callback/           ← Worker (Redis queue) → Middleware → Dispatcher → Handlers
│   ├── sse_manager.py         ← Redis Pub/Sub, Last Value Cache, Dialog Focus
│   ├── scheduler_service.py   ← 12 APScheduler задач с Redis-локами
│   └── worker_pool.py         ← DynamicWorkerPool (vk_api: max 10, db: max 20)
├── crud/                      ← Прямые DB-операции, db: Session параметром
└── schemas/                   ← Pydantic v2 модели
```

### Фронтенд (React + TypeScript + Tailwind)

```
├── App.tsx                    ← AuthContext → ProjectsProvider → Router
├── contexts/ProjectsContext   ← God-контекст: 15+ состояний, 3 под-хука
├── features/                  ← 22 модуля (auth, posts, schedule, messages, products, ...)
│   └── {module}/components/ + hooks/ + types.ts
├── services/api/              ← 27 API-модулей, все через callApi (НЕ прямой fetch!)
├── shared/                    ← Компоненты, хуки, utils, config.ts (API_BASE_URL)
└── tests/                     ← Vitest + Testing Library (~15% покрытие)
```

### Инфраструктура

| Компонент | Реализация | Защита multi-instance |
|---|---|---|
| VK Callback | Redis Queue (BRPOP) + ThreadPool(4) | Redis-лок worker_lock |
| SSE | Redis Pub/Sub + asyncio.Queue(500) | _instance_id фильтр |
| Scheduler | APScheduler, 12 задач | Redis SET NX EX для каждой задачи |
| Шифрование | Fernet (AES-128-CBC + HMAC-SHA256) | Ротация через encryption_key_new |

### 🏗️ Архитектор (architect)

**Зона:** Проектирование решений, API-контракты, схемы БД, оценка N+1.

**Когда запускать:** После аналитика, ДО написания кода.

**Шаблон промпта:**
```
Ты — архитектор проекта VK Content Planner (FastAPI + React/TypeScript + SQLAlchemy).
Задача: {описание}.

Контекст от аналитика: {результат_аналитика}

ОБЯЗАТЕЛЬНЫЕ ПРАВИЛА проекта:
- PK модели: String = str(uuid.uuid4()), НЕ auto-increment
- FK к projects: ВСЕГДА ondelete="CASCADE"
- Токены/ключи: ВСЕГДА EncryptedString (models_library/types.py)
- Новые колонки: через самописную миграцию (check_and_add_column), НЕ Alembic
- relationship с lazy="joined" ТОЛЬКО для 1:1 связей (не для списков!)
- Для списков: всегда subqueryload в запросе, НЕ в relationship
- Даты: DateTime тип, НЕ String (исторически есть String-даты, новые — только DateTime)
- JSON-данные: если нужна фильтрация — отдельная таблица; если только хранение — Text с json.loads

Спроектируй:
1. SQLAlchemy модели: таблица, поля, типы, FK, индексы, unique constraints
2. Миграция: функция в db_migrations/ (идемпотентная, check_and_add_column)
3. Pydantic-схемы: Create, Update, Response
4. API-эндпоинты: метод, путь, request, response, query params
5. Фронтенд-контракт: API-модуль, хуки, компоненты, типы
6. Анализ рисков:
   - N+1 запросы: где возможны, как предотвратить (subqueryload / joinedload)
   - Race conditions: check-then-act паттерны → использовать INSERT ON CONFLICT или db-level locks
   - Каскады: что произойдёт при удалении parent-записи
   - Конкурентность: если два пользователя редактируют одно и то же
   - Red Zone: затрагивает ли шифрование, callback, SSE, scheduler

Верни структурированный план. Для моделей и схем — готовый Python-код.
```

**Под-агенты архитектора:**
- `arch-models`: чтение models.py для проверки уникальности таблиц и паттернов
- `arch-migrations`: чтение db_migrations/ для определения формата новой миграции
- `arch-similar`: чтение аналогичного модуля для копирования паттерна

**Зона:** Python-код — модели, миграции, CRUD, сервисы, роутеры.

**Когда запускать:** После одобрения плана архитектором.

```
ПРАВИЛА КОДА:
- Комментарии — на русском
- CRUD: функции get_*, create_*, update_*, delete_*, параметр db: Session
- Commit вызывается В CRUD, не в роутере
- Для bulk-операций: synchronize_session=False в delete()

- try/except с db.rollback() при операциях с commit
- Токены → EncryptedString, НИКОГДА Text
- Параметры пользователя → ВСЕГДА валидация через Pydantic
- Не логировать токены и пароли
- background-задачи → get_background_session(), обязательно SessionLocal.remove() в finally

Аналогичный модуль для образца: {путь к файлам аналогичного модуля}
```

```
   - Бэкенд: docs/backend/ (если новые эндпоинты, модели)
   - Фронтенд: docs/frontend/ (если новые компоненты, хуки)
   - API: docs/api/ (если новые эндпоинты)
```

## Использование Sequential Thinking (MCP)

Оркестратор работает со сложными задачами, где нужно **спланировать декомпозицию** — кому что поручить, в каком порядке, какие зависимости между раундами. Для этого используй `sequential-thinking`:

### Когда применять:
- **Раунд 0** — определение масштаба и планирование раундов
- **Раунд 5** — интеграция результатов бэкенда и фронтенда (поиск конфликтов)
- **Раунд 7** — анализ ошибок из QA и выбор стратегии исправления

### Пример (Раунд 0):

```
→ sequential-thinking: thought
  "Задача: добавить функционал промокодов.
   Затронутые модули:
   - Бэкенд: модель, CRUD, сервис, роутер, миграция (5 файлов)
   - Фронтенд: типы, API, хук, 2 компонента, роутинг (6 файлов)
   - Инфра: нет новых зависимостей
   
   Масштаб: средний (11 файлов, 2 модуля)
   
   Зависимости:
   - Фронт зависит от API-контракта бэка → архитектор сначала
   - Бэк и фронт можно параллельно после контракта
   - QA после обоих
   
   Red Zone: нет (промокоды не трогают шифрование, callback, SSE)
   
   План: Раунд 1→2→4(параллельно)→5→6→8
   Пропускаем: Раунд 3 (нет сложных тестов), Раунд 9 (после одобрения)"
```

**Правило:** всегда используй sequential-thinking в Раунде 0 для задач масштаба «средний» и «крупный». Экономит токены на планировании в основном чате.

## Процедура оркестрации (10 раундов)

### Раунд 0: Приём задачи

1. Получить описание от пользователя
2. **Использовать sequential-thinking** для определения масштаба и плана раундов
3. Определить масштаб:
   - **Малый** (1 модуль, <5 файлов) → НЕ использовать оркестратор
   - **Средний** (2–3 модуля, 5–15 файлов) → Аналитик → Архитектор → Разработчики → QA
   - **Крупный** (>3 модуля, >15 файлов) → Полный цикл со всеми ролями
4. Показать план пользователю

### Раунд 1: Разведка (параллельно, 2–3 агента)

```
├── Аналитик бэкенда → JSON: файлы, экспорты, зависимости
├── Аналитик фронтенда → JSON: компоненты, хуки, API-модули
└── Аналитик безопасности → Red Zone Impact Assessment
```

### Раунд 2: Проектирование

```
└── Архитектор → план: модели, схемы, API-контракт, миграция, риски
```

> ⚠️ **СОГЛАСОВАНИЕ С ПОЛЬЗОВАТЕЛЕМ** — показать план, дождаться одобрения!

### Раунд 3: Подготовка тестов (ДО реализации)

```
└── QA → snapshot текущего состояния: какие файлы есть, какие импорты, get_errors baseline
```

Это критически важно: фиксируем «до» чтобы сравнить с «после».

### Раунд 4: Реализация (параллельно)

```
├── Бэкенд-разработчик → модель + миграция + CRUD + сервис + роутер
└── Фронтенд-разработчик → типы + API-модуль + хуки + компоненты
```

Каждый разработчик может запускать **своих под-агентов** (макс. 2 уровень):
```
Бэкенд-разработчик:
  ├── Под-агент: чтение models.py (проверка уникальности таблиц)
  ├── Под-агент: чтение аналогичного CRUD (образец паттерна)
  └── Код пишет сам разработчик

Фронтенд-разработчик:
  ├── Под-агент: чтение docs/ui-kit/ (стили и компоненты)
  ├── Под-агент: чтение аналогичного module (образец)
  └── Код пишет сам разработчик
```

### Раунд 5: Интеграция

Руководитель сверяет результаты бэкенда и фронтенда:

1. **API-контракт**: пути эндпоинтов совпадают? Имена полей совпадают?
2. **Типы данных**: Python datetime → ISO string → TypeScript string?
3. **Ошибки**: фронт обрабатывает все коды ошибок бэкенда?

При несовпадениях → запускает **Фиксера**:
```
└── Фиксер: «Бэкенд отдаёт {поле} как {тип}, фронт ожидает {тип}. Исправь {кого}»
```

### Раунд 6: Качество (параллельно)

```
├── QA → компиляция + импорты + Red Zone + API-контракт + тесты
└── UI/UX → аудит по дизайн-системе (если есть UI-изменения)
```

### Раунд 7: Исправления

Руководитель применяет исправления из Раунда 6:
- Ошибки компиляции → исправляет сам или запускает фиксера
- Red Zone нарушения → **СТОП**, доложить пользователю
- UI-нарушения → применяет патчи от UI/UX

### Раунд 8: Финальная верификация

```
└── QA (повторно) → get_errors на ВСЕХ затронутых файлах, сравнение с baseline из Раунда 3
```

Если новых ошибок нет → формируется чеклист тестирования.

> ⚠️ **СОГЛАСОВАНИЕ С ПОЛЬЗОВАТЕЛЕМ** — показать чеклист, дождаться тестирования!

### Раунд 9: Документация + DevOps (⚡ параллельно, после одобрения)

```
├── Документатор → changelog + docs
└── DevOps → зависимости + деплой (при необходимости)
```

**⚡ ПАРАЛЛЕЛЬНО:** Документатор и DevOps работают с разными файлами и не зависят друг от друга — запускать одновременно. Оба зависят от результатов Раунда 8, но не друг от друга.

## Правила координации между агентами

### 1. Передача контекста (сжатие)

Каждый раунд **сжимает** данные для следующего:

| От | К | Формат передачи |
|---|---|---|
| Аналитик → Архитектор | JSON: файлы + зависимости + Red Zone (НЕ код) |
| Архитектор → Разработчики | План: модели + API-контракт + образец (НЕ анализ) |
| Разработчики → QA | Список файлов + действия (create/modify) (НЕ весь код) |
| QA → Руководитель | Ошибки + чеклист (НЕ процесс проверки) |

### 2. Изоляция данных между агентами

**КРИТИЧНО**: Саб-агенты НЕ видят контекст друг друга. Руководитель — единственная точка координации.

```
              Руководитель (основной чат)
             /          |          \
     Аналитик    Архитектор    Разработчик
      (изолирован)  (изолирован)  (изолирован)
           |                         |
      под-агент               под-агент
      (изолирован)            (изолирован)
```

Каждый агент получает **только то, что ему нужно**:
- Аналитик: описание задачи → возвращает структуру
- Архитектор: описание + структура от аналитика → возвращает план
- Разработчик: план архитектора + образец → возвращает код

### 3. Предотвращение конфликтов при параллельной работе

Бэкенд и фронтенд работают параллельно, но могут конфликтовать:

| Конфликт | Как предотвратить |
|---|---|
| Разные имена полей в API | Архитектор фиксирует контракт ДО реализации |
| Оба правят один файл | Руководитель назначает файлы по ответственности |
| Несовместимые типы данных | QA проверяет в Раунде 6 |
| Дублирование кода в shared/ | Руководитель контролирует — один агент создаёт, другой импортирует |

### 4. Файлы с ЭКСКЛЮЗИВНЫМ владением

| Файл | Кто имеет право менять |
|---|---|
| `backend_python/models.py` | ТОЛЬКО бэкенд-разработчик |
| `backend_python/main.py` | ТОЛЬКО бэкенд-разработчик (регистрация роутера) |
| `App.tsx`, `AppContent.tsx` | ТОЛЬКО фронтенд-разработчик (роутинг) |
| `contexts/ProjectsContext.tsx` | ТОЛЬКО руководитель (слишком критично) |
| `shared/utils/apiClient.ts` | НИКТО (Red Zone) |
| `services/sse_manager.py` | НИКТО без явного одобрения пользователя |
| `services/vk_callback/*` | НИКТО без явного одобрения пользователя |

### 5. Максимальная параллелизация

| Раунд | Агенты параллельно | Макс. саб-агентов |
|---|---|---|
| 1 (Разведка) | 2–3 аналитика | Каждый запускает 0–2 под-агента |
| 4 (Реализация) | бэкенд + фронтенд | Каждый запускает 1–3 под-агента |
| 6 (Качество) | QA + UI/UX | QA запускает 2–3 под-агента |

**Лимиты:** макс. 4 саб-агента параллельно на уровне руководителя. Под-агенты — последовательно внутри родителя.

## Ограничения

- Максимум **4 саб-агента параллельно** на уровне руководителя
- Максимум **2 уровня вложенности** (руководитель → роль → под-агент)
- НЕ использовать для задач < 5 файлов — overhead не окупится
- Red Zone файлы → НИКОГДА менять без явного одобрения пользователя
- Руководитель всегда в основном чате — единственная точка координации
- После каждого раунда — краткий статус пользователю (1–2 строки)
