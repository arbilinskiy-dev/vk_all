# Arch Review — Архитектурный ревью и оптимизация

## Суть

Глубокий анализ кода на архитектурные антипаттерны: N+1 запросы, избыточные обращения к БД/API, неоптимальные структуры данных, отсутствие кэширования. Цель — найти и устранить узкие места **до** того, как они станут проблемой на масштабе.

## Процедура (6 шагов)

### Шаг 1 — Определить область ревью

Спросить пользователя (если не указано явно):
- Конкретный файл / функция / эндпоинт?
- Целая фича / модуль?
- Весь бэкенд / фронтенд?

Если область широкая — начать с самых «горячих» путей (списки, загрузка дашборда, массовые операции).

### Шаг 2 — Собрать код и построить карту запросов

Для каждой целевой функции / эндпоинта:

1. **Прочитать весь стек** — роутер → сервис → crud → модель (бэкенд) или компонент → хук → API-вызов (фронтенд)
2. **Выписать каждый запрос к БД/API** с указанием:
   - Откуда вызывается (файл:строка)
   - Внутри цикла? (пометить `⚠️ LOOP`)
   - Какие данные возвращает
   - Есть ли аналогичный запрос в другом месте стека

3. **Посчитать формулу запросов:**
```
Для списка из N объектов:
  Запросов = <фиксированных> + N × <на каждый объект>
  
Пример: загрузка проектов с постами и тегами
  Запросов = 1 (projects) + N × 1 (posts) + N × 1 (tags) + N × 1 (stats)
  При N=200: 1 + 600 = 601 запрос
```

### Шаг 3 — Анализ по чеклисту (8 пунктов)

Пройти каждый пункт и зафиксировать статус:

| # | Пункт | Что проверять |
|---|---|---|
| 1 | **N+1 запросы** | Есть ли цикл, внутри которого запрос к БД/API? |
| 2 | **Batch-возможности** | Можно ли объединить N одинаковых запросов в 1 через `IN (...)`? |
| 3 | **Prefetching** | Какие данные общие для всех объектов? Можно ли запросить заранее? |
| 4 | **Структуры данных** | Используется `array.find()` / `list` для lookup вместо `Map` / `dict`? |
| 5 | **Алгоритмическая сложность** | Есть ли O(n²) там, где можно O(n)? Вложенные циклы? |
| 6 | **Кэширование** | Повторяющиеся вычисления / запросы, которые можно закэшировать? |
| 7 | **Lazy loading** | Загружаются ли данные, которые пользователь может никогда не увидеть? |
| 8 | **Дублирование запросов** | Один и тот же запрос вызывается из разных мест стека? |

Подробнее по каждому пункту: [references/checklist_details.md](references/checklist_details.md)

### Шаг 4 — Составить отчёт

Формат обязательного отчёта:

```
═══════════════════════════════════════════════════════
  АРХИТЕКТУРНЫЙ РЕВЬЮ: <область>
═══════════════════════════════════════════════════════

📊 КАРТА ЗАПРОСОВ (текущее состояние)
  Функция: load_projects_with_details()
  Фиксированных запросов: 1
  Запросов на каждый объект: 5
  При N=200: 1 + 200×5 = 1001 запрос

🔍 НАЙДЕННЫЕ ПРОБЛЕМЫ

  ❌ #1 N+1: get_project_posts() вызывается в цикле
     Файл: services/project_service.py:45
     Масштаб: 200 лишних запросов при 200 проектах
     
  ❌ #2 Prefetch: get_all_tags() вызывается для каждого проекта,
     но теги одинаковые — можно 1 запрос вместо 200
     Файл: services/project_service.py:48

  ⚠️ #3 Структура данных: линейный поиск тега по массиву
     Файл: services/project_service.py:52
     tags.find(t => t.id === tagId) — O(n) на каждый вызов

  ✅ #4 Lazy loading — OK, данные загружаются по запросу
  ✅ #5 Кэширование — React Query с staleTime: 5min
  ...

📈 ПЛАН ОПТИМИЗАЦИИ

  Было:  1001 запрос (при 200 проектах)
  Будет: 5 запросов (3 batch + 2 индивидуальных)
  Экономия: –99.5%

  Шаги:
  1. Заменить get_project_posts() на batch_get_posts(project_ids)
  2. Вынести get_all_tags() перед циклом (1 запрос)
  3. Построить tags_map: dict[int, Tag] для O(1) lookup
  ...

═══════════════════════════════════════════════════════
```

### Шаг 5 — Реализовать оптимизации (после одобрения)

**Показать отчёт пользователю и дождаться одобрения.**

При реализации:
1. Создать batch-функции для запросов, которые были в циклах
2. Построить Map/dict для данных, к которым обращаются по ID
3. Вынести prefetch-запросы перед основным циклом
4. Проверить, что внешний контракт функций не изменился
5. Проверить типизацию (`get_errors`)

### Шаг 6 — Верификация

1. Убедиться, что функционал не изменился (те же входы → те же выходы)
2. Подсчитать финальную формулу запросов, сравнить с исходной
3. Составить чеклист тестирования для пользователя

## Когда применять автоматически (без явного запроса)

Даже без триггера «архитектурный ревью», если при написании нового кода ты видишь:
- Цикл for/forEach, внутри которого `await` запрос к БД/API → **остановись и примени Шаг 2-3**
- `array.find()` / `.filter()` внутри другого цикла → **замени на Map/dict**
- Один и тот же `fetch`/`query` вызывается из 3+ мест → **предложи единый prefetch**

Не нужно запускать полную процедуру — просто примени принцип из инструкции (раздел 5 ai_instr).

## Типовые паттерны оптимизации

### Паттерн 1: Batch query (Python/SQLAlchemy)

```python
# ❌ Было: N запросов
for project_id in project_ids:
    posts = db.query(Post).filter(Post.project_id == project_id).all()

# ✅ Стало: 1 запрос
all_posts = db.query(Post).filter(Post.project_id.in_(project_ids)).all()
posts_by_project = defaultdict(list)
for post in all_posts:
    posts_by_project[post.project_id].append(post)
```

### Паттерн 2: Prefetch + Map (TypeScript/React)

```typescript
// ❌ Было: N запросов
const results = [];
for (const project of projects) {
  const tags = await api.getTags(project.id);      // N запросов
  const stats = await api.getStats(project.id);     // ещё N
  results.push({ ...project, tags, stats });
}

// ✅ Стало: 2 batch-запроса
const projectIds = projects.map(p => p.id);
const [allTags, allStats] = await Promise.all([
  api.getTagsBatch(projectIds),       // 1 запрос
  api.getStatsBatch(projectIds),      // 1 запрос
]);
const tagsMap = new Map(allTags.map(t => [t.projectId, t]));
const statsMap = new Map(allStats.map(s => [s.projectId, s]));

const results = projects.map(p => ({
  ...p,
  tags: tagsMap.get(p.id),
  stats: statsMap.get(p.id),
}));
```

### Паттерн 3: Lookup Map вместо вложенного цикла

```python
# ❌ O(n × m) — вложенный поиск
for post in posts:
    for tag in all_tags:
        if tag.id == post.tag_id:
            post.tag_name = tag.name

# ✅ O(n + m) — Map
tag_map = {tag.id: tag.name for tag in all_tags}
for post in posts:
    post.tag_name = tag_map.get(post.tag_id, "")
```

### Паттерн 4: Кэширование справочников (React)

```typescript
// ❌ Каждый компонент запрашивает заново
const { data: tags } = useQuery(['tags'], fetchTags);  // в 5 компонентах = 5 запросов

// ✅ Один запрос через staleTime
const { data: tags } = useQuery(['tags'], fetchTags, {
  staleTime: 5 * 60 * 1000,  // 5 минут — данные считаются свежими
});
// React Query автоматически переиспользует кэш
```

## Использование Sequential Thinking (MCP)

Для сложного архитектурного анализа используй MCP-сервер `sequential-thinking`. Это позволяет AI **«думать» в отдельном изолированном контексте** — не тратя токены основного чата.

### Когда применять:

| Ситуация | Почему sequential-thinking |
|---|---|
| Анализ карты запросов (Шаг 2) при >5 эндпоинтах | Много данных, нужно пошагово считать формулы запросов |
| Сравнение нескольких вариантов оптимизации (Шаг 4) | Нужно взвесить trade-offs каждого подхода |
| Оценка рисков при масштабной рефакторинге | Каскад последствий: что сломается, если изменить X |

### Пример использования:

```
→ sequential-thinking: thought
  "Анализирую эндпоинт GET /projects. Стек: router → service → crud.
   Запросы: 1 (projects) + N×1 (posts) + N×1 (tags) = 1 + 2N.
   При N=200: 401 запрос.
   Вариант A: batch через IN — 3 запроса.
   Вариант B: joinedload — 1 запрос, но heavy response.
   Вариант A лучше для списков, B — для detail-view."

→ В основной чат: только итоговая рекомендация
```

**Правило:** используй sequential-thinking когда нужно просчитать ≥3 варианта или когда карта запросов содержит ≥10 SQL.

## Ограничения

- **Не менять бизнес-логику** — только оптимизировать способ получения/обработки данных
- **Не менять внешний контракт** — те же входы → те же выходы
- **Одобрение перед реализацией** — показать отчёт и план, дождаться ОК
- **Не оптимизировать преждевременно** — если данных < 10, оптимизация не нужна
