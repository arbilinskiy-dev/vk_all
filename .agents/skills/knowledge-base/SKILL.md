## Формат файлов

Каждый MD-файл следует единому формату:

```markdown
# Заголовок

> Краткое описание (1-2 предложения) — о чём этот файл

## Содержание
- [Секция 1](#секция-1)
- [Секция 2](#секция-2)

---

## Секция 1

Контент...

## Секция 2

Контент...

---

_Последнее обновление: ГГГГ-ММ-ДД_
_Источник: [кто дал информацию / откуда]_
_Теги: #тег1 #тег2 #тег3_
```

## Процедуры

### 1. Добавление новой информации

Пользователь скидывает текст, голосовую расшифровку, список правил, примеры и т.д.

**Шаги:**

1. **Классифицировать** — определить категорию:
   - Стандарт → `standards/`
   - Процесс/регламент → `processes/`
   - Чеклист → `checklists/`
   - Антипаттерн → `anti-patterns/`
   - Роль → `roles/`
   - Шаблон → `templates/`
   - Спека для AI → `ai-agents/`
   - Если информация пересекает несколько категорий — **разделить** на части

2. **Проверить дубли** — через саб-агент прочитать существующие файлы в целевом разделе:
   - Если тема уже покрыта файлом → **дополнить** существующий файл (не создавать новый)
   - Если есть частичное пересечение → **вынести** общую часть в отдельный файл и ссылаться
   - Если дублей нет → **создать** новый файл

3. **Структурировать** — преобразовать сырой текст:
   - Выделить ключевые тезисы
   - Убрать воду и повторы
   - Сгруппировать по логическим блокам
   - Добавить заголовки и списки
   - Сохранить конкретные примеры и формулировки пользователя
   - Добавить теги в конце файла

4. **Записать** — через `create_file` или `replace_string_in_file`:
   - Если новый файл — создать с полным форматом (см. «Формат файлов»)
   - Если обновление — прочитать текущий файл через `read_file`, добавить информацию, обновить дату

5. **Обновить индекс** — добавить/обновить строку в `_meta/index.md`

6. **Отчитаться** — сообщить пользователю:
   - Куда записано (файл, раздел)
   - Что нового добавлено (кратко)
   - Были ли совпадения/дополнения к существующему

### 2. Поиск и просмотр информации

Пользователь спрашивает «что есть в базе знаний по теме X?» или «покажи стандарты».

**Шаги:**

1. Прочитать `_meta/index.md` — найти релевантные файлы по тегам/описанию
2. Прочитать нужные файлы через `read_file`
3. Сформировать компактный ответ с ссылками на файлы

### 3. Проверка на дубли и реструктуризация

Пользователь просит «проверь базу на дубли» или «реструктурируй».

**Шаги:**

1. ⚡ **Саб-агент: Сканирование** — прочитать ВСЕ файлы из всех разделов
2. **Анализ пересечений:**
   - Найти одинаковые тезисы в разных файлах
   - Найти файлы, которые должны быть в другом разделе
   - Найти файлы, которые стоит объединить или разделить
3. **Отчёт** — показать пользователю план изменений
4. **Выполнить** после подтверждения — переместить, объединить, вычистить дубли
5. **Обновить индекс**

### 4. Создание нового раздела

Если поступает информация, не подходящая ни под одну категорию:

1. Предложить пользователю название нового раздела
2. Создать папку с первым файлом
3. Обновить README.md и индекс
4. Обновить структуру в этом SKILL.md (через редактирование файла)

## Делегирование саб-агентам

Следующие шаги выполнять через `runSubagent` для экономии контекста:

| Задача | Промпт |
|--------|--------|
| Проверка дублей | «Прочитай все MD-файлы в папке agency-knowledge-base. Для каждого файла верни: путь, теги, ключевые тезисы (по 1 строке на тезис). Формат — JSON-массив.» |
| Сканирование раздела | «Прочитай все файлы в папке agency-knowledge-base/{раздел}/. Верни содержимое каждого файла.» |
| Анализ пересечений | «Вот содержимое файлов базы знаний: {данные}. Найди: (1) дублирующие тезисы в разных файлах, (2) файлы в неправильных разделах, (3) файлы для объединения. Верни таблицу.» |

## Правила

1. **Не дублировать.** Одна единица знания — в одном месте. Ссылки вместо копирования.
2. **Атомарность файлов.** Один файл — одна тема. Если файл >300 строк — рассмотреть разделение.
3. **Теги обязательны.** Каждый файл заканчивается строкой `_Теги: #тег1 #тег2_` для навигации.
4. **Индекс актуален.** После любого изменения — обновить `_meta/index.md`.
5. **Сохранять формулировки.** Конкретные примеры и фразы от пользователя сохранять дословно (выделять цитатами `>`).
6. **Не выдумывать.** Записывать только то, что предоставил пользователь. Структурировать — да, додумывать — нет.
7. **Без лишних инструментов.** Все файлы — через стандартные `read_file`, `create_file`, `replace_string_in_file`. Никаких GitHub MCP.
