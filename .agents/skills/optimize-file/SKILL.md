# Optimize File — Декомпозиция монолитных файлов

## Суть

Превратить большой файл в **тонкий хаб** (контейнер), а логику разложить по отдельным файлам.
Результат: прежний функционал, визуал, тексты, внешний контракт (имя + props/сигнатура) — но код модульный и читаемый.

## Процедура (7 шагов)

### Шаг 1 — Анализ файла

1. Прочитать файл целиком
2. Определить тип: **React-компонент** (.tsx) или **Python-модуль** (.py)
3. Подсчитать строки. Если < 150 строк — сообщить пользователю, что декомпозиция избыточна
4. Найти все внешние импорты и экспорты — это **внешний контракт**, который нельзя ломать
5. Найти все места, где этот файл используется (`list_code_usages`)

### Шаг 2 — Классификация блоков

Разбить содержимое файла на логические блоки. Для каждого определить категорию:

**React (.tsx):**
| Категория | Куда выносить | Пример |
|---|---|---|
| Стейт + логика | `useXxxLogic.ts` хук | useState, useEffect, обработчики, вычисления |
| Подкомпоненты | `XxxSection.tsx` | JSX-блоки > 30 строк с собственной логикой |
| Утилиты/хелперы | `xxxUtils.ts` | Чистые функции без React |
| Типы/интерфейсы | `types.ts` | Локальные типы, используемые в нескольких файлах |
| Константы | `constants.ts` | Массивы опций, конфиги, маппинги |

**Python (.py):**
| Категория | Куда выносить | Пример |
|---|---|---|
| CRUD/DB операции | `xxx_crud.py` | Функции с session, запросы к БД |
| Бизнес-логика | `xxx_service.py` | Обработка данных, валидация, трансформации |
| API-вызовы | `xxx_client.py` | Запросы к внешним API |
| Утилиты | `xxx_utils.py` | Чистые функции |
| Схемы | `xxx_schemas.py` | Pydantic-модели |

### Шаг 3 — План декомпозиции

**Предложить план пользователю и дождаться одобрения.** План должен содержать:

```
📁 Текущий файл: features/xxx/components/BigComponent.tsx (450 строк)
   ↓ После декомпозиции: ~80 строк (хаб-контейнер)

📄 Новые файлы:
  1. features/xxx/hooks/useBigComponentLogic.ts (~200 строк)
     — стейт, эффекты, обработчики
  2. features/xxx/components/SectionA.tsx (~90 строк)
     — JSX-блок секции A
  3. features/xxx/components/SectionB.tsx (~80 строк)
     — JSX-блок секции B

🔒 Внешний контракт (не меняется):
  — Имя: BigComponent
  — Props: { projectId, onClose, isOpen }
  — Экспорт: named export
```

### Шаг 4 — Создание извлечённых файлов

Для каждого файла из плана:
1. Создать файл с правильными импортами (относительные пути, без алиасов)
2. Перенести логику **без изменений** — copy-paste, не переписывать
3. Экспортировать всё, что нужно хабу

**Для хуков** — следовать паттерну `{ state, actions }`:
```typescript
// Возвращаемый объект хука
return {
  state: { items, isLoading, error, selectedId, ... },
  actions: { handleSave, handleDelete, handleSelect, ... }
};
```

Подробнее: прочитать [references/hub_pattern.md](references/hub_pattern.md)

### Шаг 5 — Превращение исходного файла в хаб

Исходный файл становится тонким контейнером:
1. Импортирует хук и подкомпоненты
2. Вызывает хук, деструктурирует `{ state, actions }`
3. Рендерит подкомпоненты, прокидывая нужные props
4. **Имя и props компонента остаются прежними**

Подробнее: прочитать [references/hub_pattern.md](references/hub_pattern.md)

### Шаг 6 — Верификация

**Обязательный шаг.** Прочитать [references/verification.md](references/verification.md) и выполнить:

1. **Контракт**: имя и props/сигнатура файла не изменились
2. **Импорты**: все внешние файлы, которые импортировали исходный — работают без изменений
3. **Типы**: запустить `get_errors` на всех затронутых файлах
4. **Полнота**: ни один блок кода не потерян, не обрезан, не переписан
5. **Циклы**: нет циклических импортов между новыми файлами

### Шаг 7 — Чеклист тестирования для пользователя

Составить пошаговый чеклист: что открыть, что нажать, какой результат ожидать.
Покрыть **весь** функционал, который был в исходном файле.

Подробнее: прочитать [references/verification.md](references/verification.md)

## Организация файловой структуры (опциональный шаг)

**Когда применять:** если после декомпозиции (или до неё) в папке `components/` или `hooks/` модуля **>15 файлов** на одном уровне — предложить пользователю разбить на подпапки.

### Процедура

1. **Анализ зависимостей** — для каждого файла определить, кто кого импортирует (`grep_search` по импортам)
2. **Группировка** — выделить кластеры файлов по функциональности:
   - Файлы, которые импортируют друг друга, попадают в одну группу
   - Общие утилиты/типы остаются в корне или выносятся в `shared/`
   - Примеры групп: `chat/`, `stats/`, `conversations/`, `user-info/`, `attachments/`
3. **План** — предложить пользователю структуру:
   ```
   components/
     chat/          — ChatView, ChatInput, ChatMessage, ...
     stats/         — StatsPage, StatsChart, StatsTable, ...
     conversations/ — ConversationsSidebar, ConversationItem, ...
   hooks/
     chat/          — useMessageHistory, useChatSend, ...
     stats/         — useStatsChartLogic, ...
   ```
4. **Перемещение** — после одобрения:
   - Создать подпапки (`create_directory`)
   - Переместить файлы (`Move-Item` в терминале)
   - **Не переименовывать файлы** — только менять расположение
5. **Фикс импортов** — обновить все `import` пути:
   - Внутри перемещённых файлов (ссылки на соседей и на другие группы)
   - В файлах-потребителях снаружи модуля
   - В тестах: обновить `vi.mock(...)` строковые пути
   - Проверить реэкспорты из `index.ts` (если есть)
6. **Верификация** — `get_errors` на всех файлах + `npx tsc --noEmit`

### Правила группировки

- **Минимум 3 файла** в подпапке — иначе не выносить
- **Корень `components/`** — оставить файлы-контейнеры верхнего уровня (точки входа модуля)
- **Именование папок** — kebab-case, по домену (не по типу файла)
- **Хуки следуют за компонентами** — если `components/chat/` содержит `ChatView`, то `hooks/chat/` содержит `useChatLogic`
- **Не вкладывать глубже 1 уровня** — `components/chat/`, но НЕ `components/chat/input/`

## Ограничения

- **Не переписывать логику** — только перемещать блоки кода as-is
- **Не менять внешний контракт** — имя, props, экспорт остаются прежними
- **Не создавать лишних абстракций** — простой вынос, не обобщение
- **Относительные пути** — без алиасов вроде `@/`
- **Одобрение перед началом** — план декомпозиции требует ОК от пользователя

## Делегирование саб-агентам

Для экономии контекстного окна следующие шаги рекомендуется выполнять через `runSubagent`:

| Шаг | Что делегировать | Промпт для саб-агента |
|---|---|---|
| Шаг 1 — Анализ файла | Чтение файла + подсчёт строк + поиск импортов/экспортов + list_code_usages | «Проанализируй файл {путь}. Верни JSON: fileType, lineCount, imports[], exports[], usedBy[], blocks[{startLine, endLine, category, description}]» |
| Шаг 2 — Классификация | Разбиение на логические блоки с категориями | Объединить с Шагом 1 в одного саб-агента |
| Шаг 4 — Reference-файлы | Чтение hub_pattern.md, hook_extraction.md, python_decomposition.md | «Прочитай reference-файлы и для каждого блока [{список}] сгенерируй готовый код файла. Верни массив [{fileName, code}]» |
| Шаг 6 — Верификация | Проверка контракта, импортов, типов, полноты | «Проверь декомпозицию: контракт {props}, созданные файлы {список}, потребители {usedBy}. Запусти get_errors. Верни {passed, issues[]}» |
| Организация папок — Зависимости | Граф импортов между файлами в папке | «Проанализируй зависимости в папке {путь}. Верни граф: {файл: [импортирует]}» |
| Организация папок — Фикс | Обновление import-путей после перемещения | «Файлы перемещены: {список}. Найди ВСЕ файлы с импортами старых путей. Верни [{file, oldImport, newImport}]» |

**⚡ ПАРАЛЛЕЛЬНО:** Шаг 1+2 уже объединены в одного агента — ОК. При декомпозиции на >2 файлов: генерация кода разных модулей (Шаг 4) может выполняться параллельно (2–3 агента генерируют разные файлы одновременно), если блоки не имеют взаимных импортов. Верификация (Шаг 6) — только после завершения всех.
