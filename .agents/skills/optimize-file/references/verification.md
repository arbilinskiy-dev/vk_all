# Verification — Верификация после декомпозиции

## Автоматические проверки (AI выполняет сам)

### 1. Проверка типов

```
get_errors на все затронутые файлы: хаб + все новые файлы + все импортёры исходного файла
```

Ноль ошибок = проходит. Любая ошибка = исправить перед тем, как двигаться дальше.

### 2. Проверка внешнего контракта

```
list_code_usages для имени компонента/функции
```

Все места использования **не должны требовать изменений**. Если хоть одно требует — контракт нарушен.

### 3. Проверка полноты кода

Прочитать хаб + все извлечённые файлы. Сверить со **снимком исходного файла** (сделанным до рефакторинга):
- Каждый `useState` присутствует
- Каждый `useEffect` присутствует
- Каждый обработчик (`handleXxx`, `onXxx`) присутствует
- Каждый JSX-блок присутствует
- Каждый условный рендеринг присутствует
- Ни одна строка не удалена и не переписана

### 4. Проверка циклических импортов

Граф импортов новых файлов не должен содержать циклов:
```
A → B → A  ← ЦИКЛ, запрещено
A → B, A → C, B → C  ← OK
```

### 5. Проверка путей импорта

- Все импорты — **относительные** (`../hooks/useXxx`, не `@/hooks/useXxx`)
- Пути ведут к реальным файлам (проверить `file_search`)
- Расширения не указаны в импортах (.ts, .tsx опускаются)

## Чеклист тестирования для пользователя

AI составляет этот чеклист после завершения рефакторинга. Формат:

```markdown
## Чеклист тестирования: {Название компонента}

### Основной функционал
- [ ] Открыть {страницу/модалку/раздел}
- [ ] Проверить, что {элемент} отображается корректно
- [ ] Нажать {кнопку} → ожидаемый результат: {что должно произойти}
- [ ] Ввести данные в {поле} → ожидаемый результат: {валидация/сохранение}

### Граничные случаи
- [ ] Пустое состояние (нет данных) → отображается {placeholder/skeleton}
- [ ] Ошибка сервера → отображается {toast/сообщение об ошибке}
- [ ] Быстрые повторные нажатия → не дублируются запросы

### Визуальная целостность
- [ ] Скроллбары работают ({область с прокруткой})
- [ ] Адаптивность на узком экране
- [ ] Все иконки, цвета, отступы — как прежде
```

### Правила составления чеклиста

1. **Покрыть 100% функционала** — каждый обработчик из исходного файла должен проверяться хотя бы одним пунктом
2. **Конкретные действия** — «нажать кнопку Сохранить», не «проверить сохранение»
3. **Ожидаемый результат** — для каждого пункта указать, что пользователь должен увидеть
4. **Граничные кейсы** — пустые данные, ошибки, множественные клики
5. **Визуал** — убедиться, что ничего не «поехало»
