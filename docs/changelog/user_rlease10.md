## 🆕 Новые возможности

### Оптимизация синхронизации статистики сообщений

**Что добавили:**  
Синхронизация статистики из лога callback-уведомлений теперь работает в фоновом режиме с отображением прогресса. Ранее при большом количестве записей (10–30 тысяч) синхронизация могла зависать или завершаться ошибкой по таймауту.

**Как использовать:**  
На странице «Мониторинг сообщений» нажмите кнопку «Синхронизация из логов». Появится прогресс-бар, показывающий две фазы: чтение логов и сохранение данных. Можно продолжать работу с другими разделами — синхронизация идёт в фоне.

**Что теперь можно:**  
Синхронизировать статистику при любом количестве записей в логах без зависаний и таймаутов. Прогресс виден в реальном времени.

## ✅ Решённые задачи

### Фильтры «Наши сообщения» / «Пользовательские» показывали пустой список

**Проблема:**  
При переключении фильтра направления в чате на «Наши сообщения» или «Пользовательские сообщения» список оставался пустым, хотя сообщения были загружены и хранились в базе.

**Решение:**  
Устранена проблема с одновременными запросами: теперь при смене фильтра устаревшие ответы корректно отбрасываются и не перезаписывают актуальные данные. Дополнительно усилена надёжность фильтрации на сервере — исключены редкие сценарии, когда фильтр мог не найти сообщения из-за особенностей хранения.

---

### Зависание страницы мониторинга при выборе «Всё время»

**Проблема:**  
При выборе фильтра периода «Всё время» на странице мониторинга сообщений страница полностью зависала — график пытался отрисовать тысячи точек данных за всю историю, что перегружало браузер.

**Решение:**  
График теперь автоматически оптимизирует отображение: при большом объёме данных переключается с часового на дневной формат, а если точек слишком много — группирует соседние значения. Страница работает плавно при любом периоде. При агрегации данных отображается соответствующий индикатор.

---

### Новые пользователи из callback-уведомлений теперь сохраняются в базу

**Проблема:**  
Когда пользователь разрешал или запрещал сообщения от сообщества, а его ещё не было в базе данных — информация терялась. Пользователь не появлялся в списке рассылки проекта.

**Решение:**  
Теперь при любом действии пользователя (разрешил / запретил сообщения) он автоматически добавляется в базу данных и в рассылку проекта. Детальная информация (имя, фото, город) подгружается автоматически ночью.

---

### Автоматическое обогащение профилей пользователей

**Что добавили:**  
Новая ежедневная задача, которая автоматически подгружает информацию о новых пользователях из ВКонтакте: имя, фамилию, пол, фото, город и другие данные.

**Как это работает:**  
Задача запускается каждую ночь в 03:00 по Москве. Она находит всех пользователей в базе, у которых ещё нет подробной информации (только ID), и запрашивает их данные через VK API. Используется сервисный ключ приложения.

**Что теперь можно:**  
В списках рассылки и мониторинге сообщений все пользователи отображаются с именами и фотографиями, даже если они появились через callback-уведомления, а не через ручную загрузку.

### Исправлена авторизация через ВКонтакте

**Проблема:**  
Авторизация через ВКонтакте могла не работать из-за устаревшего адреса перенаправления, который был прописан вручную и не обновлялся при смене окружения.

**Решение:**  
Адрес перенаправления теперь определяется автоматически на основе текущего адреса сайта. Авторизация корректно работает на всех окружениях: локальном, тестовом и рабочем.

### Кнопка обновления статистики историй не работала

**Проблема:**  
При нажатии кнопки «Обновить статистику» на странице историй ничего не происходило — статистика показывала 0 обновлённых записей. Проблема возникала в проектах, где были ручные истории (опубликованные напрямую через VK, не через автоматизацию).

**Решение:**  
Исправлена ошибка сохранения ручных историй в базу данных — каждая история теперь имеет уникальный идентификатор. Ранее все ручные истории записывались с одинаковым служебным ID, что приводило к конфликту при попытке сохранить вторую и последующие ручные истории. Также усилена обработка ошибок — теперь сбой при сохранении одной истории не блокирует остальные операции.

### Модульная архитектура сверки сообщений

**Что улучшили:**  
Механизм сверки статистики сообщений переработан для повышения надёжности. Внутренний код разбит на специализированные модули, добавлена точная отчётность — теперь прогресс-бар показывает корректное количество обработанных пользователей, даже если часть из них завершилась с ошибкой.

**Что теперь можно:**  
Сверка стабильнее работает при большом количестве проектов и диалогов. Прогресс отображается точнее: если часть диалогов не удалось обработать, это отразится в итоговой статистике.

### Сверка «Всё время» обрабатывала лишние диалоги

**Проблема:**  
При запуске сверки сообщений с фильтром «Всё время» система обрабатывала ~34 000 диалогов вместо ~500 реально активных. Процесс занимал десятки минут.

**Решение:**  
Исправлена логика выборки пользователей — теперь при любом фильтре периода (включая «Всё время») используется единый оптимальный путь. Время сверки сократилось с ~40 минут до ~3 минут.
