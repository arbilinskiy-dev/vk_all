## 1. Контекст и цель

### Бизнес-задача

Заменить живых менеджеров на AI-чатбота в диалогах VK-сообществ. 200 проектов (доставка еды), до 50 000 пользователей с перепиской в каждом. Менеджер стоит ~150к/мес. AI-система должна обрабатывать 80%+ обращений автоматически.

### Текущее состояние

- **Модуль сообщений уже существует** — полнофункциональный: VK Callback, SSE-реалтайм, кэширование в БД, UI менеджера с шаблонами.
- **Gemini API уже интегрирован** — для генерации постов, коррекции текста, маркетинга. Есть клиент с ротацией ключей, прокси, фоллбэком моделей.
- **AI для сообщений — НЕТ.** Никакой интеграции нейронки с модулем сообщений не существует.

### Существующие таблицы для сообщений

| Таблица | Назначение |
|---|---|
| `cached_messages` | Кэш каждого VK-сообщения (текст, from_id, is_outgoing, payload_json, attachments_json) |
| `project_dialogs` | Диалог пользователя с сообществом (метаданные, last_message_date, can_write) |
| `message_cache_meta` | Метаданные синхронизации |
| `message_read_status` | Статус прочтения менеджерами |
| `message_stats_hourly` | Часовые агрегаты нагрузки |
| `message_templates` | Шаблоны быстрых ответов |

### Как отличить кнопку от текста

- `payload_json IS NOT NULL` → нажатие кнопки
- `payload_json IS NULL` → текстовый ввод живого человека
- `is_outgoing = false` → входящее от пользователя
- `is_outgoing = true` → исходящее от сообщества

## 3. Архитектура AI-агента: 3 слоя

### Слой 1: База знаний (хранится внутри проекта)

Каждый проект (сообщество) имеет СВОЮ базу знаний:

| Компонент | Что хранит | Откуда берётся |
|---|---|---|
| **FAQ-пары** | Вопрос → ответ | Автоизвлечение из диалогов + ручное заполнение |
| **Каталог товаров** | Название, цена, состав, вес | Таблица `products` (уже есть) + VK Market API |
| **Бизнес-правила** | Адрес, график, условия доставки, оплата | Автоизвлечение из диалогов + посты сообщества |
| **Тон общения** | На ты/вы, эмодзи, стиль | Генерация из `generate_brand_tone()` или из ответов менеджеров |
| **Стоп-фразы** | Что AI запрещено говорить | Ручное заполнение |

#### Новые таблицы БД

**`project_ai_config`** — настройки AI для проекта:

| Поле | Тип | Назначение |
|---|---|---|
| `project_id` | FK | Ссылка на проект |
| `ai_mode` | enum | `off` / `suggest` / `auto` |
| `brand_voice_prompt` | text | Системный промпт тона бренда |
| `business_rules_text` | text | Правила бизнеса |
| `stop_phrases_json` | JSON | Запреты для AI |
| `escalation_keywords_json` | JSON | Слова-триггеры эскалации |
| `max_context_messages` | int | Сколько сообщений в контекст (default: 20) |
| `confidence_threshold` | float | Минимальная уверенность для автоотправки |

**`project_faq`** — FAQ-пары с семантическим поиском:

| Поле | Тип | Назначение |
|---|---|---|
| `id` | UUID | PK |
| `project_id` | FK | Ссылка на проект |
| `question` | text | Типичный вопрос |
| `answer` | text | Ответ (с поддержкой переменных) |
| `category` | string | Категория |
| `embedding_vector` | vector | Для pgvector семантического поиска |

### Слой 2: AI-движок (агентный цикл)

Встраивается в существующий `message_new` callback. Подробности — в разделах 4, 5, 7.

### Слой 3: Управление (три режима)

| Режим | Поведение |
|---|---|
| **Выключен** | Как сейчас, без AI |
| **Подсказчик** | AI генерирует ответ → менеджер одобряет/редактирует/отклоняет |
| **Автопилот** | AI отвечает сам, менеджер видит лог и может вмешаться |

## 4. Как работает API нейронки: Function Calling и Agentic Loop

### Фундаментальные истины (выводы из обсуждения)

**Истина 1: API без памяти. Полностью.**

Каждый запрос к API Gemini/Claude — чистый лист. «Память» реализуется тем, что **приложение каждый раз отправляет ВСЮ историю чата заново**:

```
Запрос 1: messages: [{user: "Привет"}]
Запрос 2: messages: [{user: "Привет"}, {assistant: "Привет!"}, {user: "Сколько стоит?"}]
Запрос 3: messages: [{user: "Привет"}, {assistant: "Привет!"}, {user: "Сколько стоит?"}, {assistant: "490₽"}, {user: "А с доставкой?"}]
```

Нейронка «помнит» потому что ей каждый раз пересылают весь разговор. Это делает наш код, не нейронка.

**Истина 2: AI не вызывает инструменты. AI ПРОСИТ их вызвать.**

AI — текстовая модель, она не делает HTTP-запросы и не лазит в БД. Процесс:
1. Мы отправляем запрос + **описания** инструментов
2. AI решает что ему нужны данные и возвращает `functionCall: {name: "search_products", args: {query: "Филадельфия"}}`
3. **Наш Python-код** выполняет запрос к БД
4. Мы отправляем **второй** запрос к API с результатами
5. AI теперь видит данные и генерирует текстовый ответ

**Истина 3: AI видит только то, что мы ему передали.**

AI не «видит всю БД». Он видит: system prompt + описания tools + историю диалога + результаты вызванных инструментов. Всё.

### Агентный цикл (Agentic Loop)

```python
async def process_message(project_id, user_id, message_text):
    # 1. Лёгкий контекст (всегда, ~500 слов)
    ai_config = get_project_ai_config(project_id)
    system_prompt = build_system_prompt(ai_config)  # тон + правила + стоп-фразы
    
    # 2. История из cached_messages (последние 15-20 сообщений)
    history = get_dialog_history(project_id, user_id, count=20)
    
    messages = [
        {"role": "system", "content": system_prompt},
        *history,
        {"role": "user", "content": message_text},
    ]
    
    # 3. Агентный цикл (до 5 итераций)
    for iteration in range(5):
        response = await gemini_api.generate(
            messages=messages,
            tools=TOOLS,
            tool_choice="auto",
        )
        
        if response.tool_calls:
            for tool_call in response.tool_calls:
                result = await execute_tool(tool_call.name, tool_call.args, project_id, user_id)
                messages.append({"role": "tool", "content": json.dumps(result)})
            continue  # AI получит результаты и решит: ещё tool или текстовый ответ
        
        return {"reply": response.text, "confidence": response.confidence, "tools_used": [...]}
    
    return {"reply": None, "escalate": True, "reason": "too_complex"}
```

## 5. Набор инструментов (Tools): какие и почему

### Принцип: инструменты дают ДАННЫЕ, думает — AI

AI уже знает что такое «постное», «аллерген», «лук — это ингредиент». Не нужны специальные инструменты `search_lenten_products` или `filter_by_allergen`. Нужен один `search_products` — AI получит составы и **сам** отфильтрует.

### Оптимальный набор: 8-10 инструментов

| # | Инструмент | Что даёт AI | Когда вызывает |
|---|---|---|---|
| 1 | `search_products` | Товары с составами, ценами, весом | Вопрос о блюде, цене, составе, аллергенах, постном/вегетарианском |
| 2 | `get_product_details` | Детали одного товара (фото, полное описание) | Уточнение по конкретному товару |
| 3 | `search_faq` | Ответы на типовые вопросы (семантический поиск, pgvector) | Вопрос о доставке, оплате, возврате, акциях, бонусах |
| 4 | `check_delivery_zone` | Доставляем? Цена, время, мин. заказ | Пользователь назвал адрес |
| 5 | `get_dialog_history` | Предыдущие сообщения (контекст) | Нужен контекст разговора |
| 6 | `get_user_profile` | Имя, город, история заказов, бонусы | Обращение по имени, проверка заказов |
| 7 | `get_promo_info` | Проверка промокода, текущие акции | Промокод или вопрос об акциях |
| 8 | `get_business_info` | График, адреса, телефоны, оплата | FAQ-вопросы о компании |
| 9 | `escalate_to_manager` | Передаёт менеджеру | Жалобы, сложные кейсы, агрессия |
| 10 | `create_order_draft` | Создаёт черновик заказа (фаза 4) | Пользователь хочет оформить заказ |

### Почему не больше

```
3-5 инструментов:   AI не может получить нужные данные → плохие ответы
6-10 инструментов:  Покрывает 95% кейсов, AI не путается
15-20 инструментов: AI путается какой вызвать, медленнее, дороже
```

## 6. От чего зависит точность: 4 фактора

| # | Фактор | Влияние | Сложность |
|---|---|---|---|
| 1 | **System prompt** (кто ты, правила, запреты) | 35% | Текст, итеративно |
| 2 | **Описания инструментов** (когда, зачем, с какими параметрами) | 30% | Текст, итеративно |
| 3 | **Формат данных от инструментов** (что возвращаем AI) | 20% | Код |
| 4 | **Контекст диалога** (последние N сообщений) | 10% | Код |
| 5 | Выбор модели (Gemini Pro/Flash) | 5% | Конфиг |

### Ключевые правила

**System prompt:** Конкретный, с точными правилами и запретами (~500 слов). НЕ общий «будь вежливым», а «обращайся на ты, доставка 200₽, никогда не обещай быстрее 40 минут».

**Описания инструментов:** Каждый = инструкция новому сотруднику: «когда использовать, когда НЕ использовать, что передавать».

**Формат данных:** Возвращать чистую структуру (`{name: "Филадельфия", price: "490₽", ingredients: "..."}`) а не сырой дамп из БД с 20 служебными полями.

**Контекст:** Последние 15-20 сообщений. Если пауза >24 часов — новая «сессия».

**90% успеха — это тексты (промпт + описания + формат), а не сложный код.** Итеративный тюнинг на реальных данных.

## 7. Полный пример цепочки запросов

### Кейс

Пользователь: «Добрый день, подскажите, я хочу сделать заказ, но какие блюда постные? И у меня аллергия на лук. И сколько по времени и возите ли вы на адрес г. Белгород ул. Костюкова д 38 кв 190»

4 намерения в одном сообщении: постные блюда + аллергия на лук + время доставки + проверка адреса.

### Цепочка

**HTTP-запрос №1 → Gemini API:**
- System prompt (~500 слов): тон, правила, запреты
- Tools: 6 инструментов с описаниями (~400 слов)
- Contents: сообщение пользователя

**HTTP-ответ №1 ← Gemini API:**
```json
functionCall: [
  {name: "search_products", args: {query: "все", limit: 50}},
  {name: "check_delivery_zone", args: {address: "г. Белгород, ул. Костюкова, д. 38"}}
]
```
AI попросил 2 инструмента параллельно. Не текст — а запрос на данные.

**Между запросами — наш Python-код:**
- `search_products` → SQL к таблице `products` → 18 товаров с составами
- `check_delivery_zone` → проверка зоны → `{delivers: true, time: "50-70 мин", fee: 200₽}`

**HTTP-запрос №2 → Gemini API:**
- Всё то же + добавлены `functionResponse` с результатами (18 товаров + зона)

**HTTP-ответ №2 ← Gemini API (финальный):**

AI **сам** проанализировал 18 составов, нашёл постные (без мяса/рыбы/молочки), исключил с луком, отметил спорные, добавил инфо о доставке:

> «Добрый день! 😊 Постные блюда без лука: Ролл с огурцом — 250₽, Ролл Овощной — 280₽, Салат Чука — 220₽, Эдамаме — 190₽. ⚠️ Ролл с авокадо (290₽) постный, но содержит лук зелёный. На ул. Костюкова доставляем! 50-70 минут, 200₽ (бесплатно от 1000₽).»

### Экономика

```
Наивный подход (всё в промпт):     Агентный подход (tools):
Токенов: ~20 000                     Токенов: ~1 500
Цена: ~$0.06 за сообщение           Цена: ~$0.005
Время: 5-10 секунд                  Время: 1-3 секунды
Точность: низкая (путается)         Точность: высокая
```

## 8. Полный pipeline: от данных до продукта (8 фаз)

### Обзор фаз

```
Фаза 1: Сбор данных ────────── один раз, скрипт
Фаза 2: Классификация ──────── один раз, нейронка
Фаза 3: Карта намерений ─────── один раз, аналитика
Фаза 4: Проектирование ─────── один раз, архитектура
Фаза 5: Автозаполнение БЗ ──── для каждого проекта, скрипт + нейронка
Фаза 6: Синтетическое тестирование ── per project
Фаза 7: Пилот на 1 проекте ─── live тест
Фаза 8: Масштабирование ────── 10 → 50 → 200 проектов
```

### Фаза 1: Сбор данных

Скрипт тянет ВСЕ диалоги из всех проектов. Затем **автоматическая фильтрация мусора** (без нейронки):
- ✗ Диалоги с 1 сообщением
- ✗ Только кнопки (payload)
- ✗ Текст < 3 символов без других сообщений
- ✗ Старше 2 лет
- ✗ Спам-паттерны

50 000 диалогов → фильтр → ~5 000-8 000 качественных. Их и анализируем.

### Фаза 2: Классификация намерений

Прогоняем через нейронку батчами (sub-agents). Для КАЖДОГО сообщения определяем:
- `intent` — главное намерение (ORDER_NEW, PRODUCT_QUESTION, COMPLAINT, etc.)
- `sub_intents` — дополнительные
- `required_data` — какие данные нужны были для ответа
- `complexity` — simple / medium / complex

**Решение:** Классифицировать отдельные сообщения (не сессии), потому что на этом этапе фиксируем обращения, не решения. Сессии — на фазе 5.

### Фаза 3: Карта намерений

Агрегация результатов классификации. Статистическая картина: какие типы обращений, % каждого, средняя сложность, автоматизируемость. Из `required_data` выводим список инструментов.

### Фаза 4: Проектирование агента

На основе фазы 3 — точно знаем (не гадаем) какие инструменты нужны. Проектируем system prompt, формат ответов, правила эскалации.

### Фаза 5: Автозаполнение базы знаний

Для КАЖДОГО проекта:
- **Из ответов менеджеров** → адреса, телефоны, график, зоны, условия, тон
- **Из товаров сообщества** → каталог с составами
- **Из постов** → акции, новинки, объявления
- Заполняем `project_ai_config`, `project_faq`, `products`, `delivery_zones`

### Фаза 6: Синтетическое тестирование

Три скилла:
1. **Генератор (attacker)** — создаёт 200 реалистичных запросов (простые, средние, сложные, edge cases, цепочки)
2. **AI-ассистент (defender)** — отвечает через реальные инструменты и БД проекта
3. **Оценщик (judge)** — оценивает по 6 критериям (точность, полнота, тон, инструменты, эскалация, безопасность), 0-10 каждый

Итерируем до среднего балла 9.0+.

### Фаза 7: Пилот на 1 проекте

Режим «подсказчик». Метрики: % одобренных, % отредактированных, % отклонённых. Цель: >80% одобренных без правок.

### Фаза 8: Масштабирование

Пилот > 80% → раскатка: 10 → 50 → 200 проектов.

## 9. Self-learning: автономная система с самообучением

### Жизненный цикл проекта (полная автономность)

```
Подключение → Обучение → Работа → Самообучение (бесконечный цикл)
```

**Шаг 1: Подключение (автоматически при получении токена)**
1. Получить данные сообщества (название, контакты, описание)
2. Скачать товары / услуги
3. Скачать ВСЕ диалоги
4. Скачать посты за последние 6 месяцев

**Шаг 2: Обучение (автоматическое наполнение тулзов)**
1. Фильтрация мусорных диалогов
2. Извлечение констант (адреса, телефоны, график) из ответов менеджеров
3. Извлечение данных из постов и описания сообщества
4. Наполнение всех тулзов данными

**Шаг 3: Работа (AI как менеджер)**
- Входящее сообщение → быстрый фильтр (мусор/кнопка/текст) → AI-агент → роутинг (автоответ / подсказка / эскалация)

**Шаг 4: Самообучение (три петли)**

### Петля А: Эскалация → менеджер решил → учимся

AI не справился → менеджер ответил → аналитик в фоне определяет почему:
- Нет данных в тулзе → **автоматически дополнить FAQ из ответа менеджера**
- Промпт неточный → добавить правило в `business_rules`
- Слишком сложно → норма, эскалация оправдана

### Петля Б: Оценка пользователя → учимся

AI ответил → спрашиваем юзера (кнопки 1-10):
- 8-10: ОК
- 5-7: «Что можно улучшить?» → юзер пишет → аналитик корректирует FAQ/промпт
- 1-4: Эскалация + Петля А

### Петля В: Пассивное (фоновый процесс, постоянно)

- Новый пост в сообществе → парсим → обновляем тулзы (акция, смена графика)
- Менеджер отредактировал ответ AI → фиксируем дельту → корректируем промпт
- 10+ похожих эскалаций за неделю → отчёт клиенту: «15 юзеров спросили про безглютеновое меню, а в вашей БД этих данных нет»

### Админ-режим (управление через чат)

Клиент общается с ботом-администратором (не с ботом-продавцом):

```
Клиент: "Покажи что лежит в FAQ про доставку"
→ Показывает записи

Клиент: "Обнови время доставки на 30-50 минут"
→ Обновляет FAQ + system_prompt

Клиент: "Покажи статистику за неделю"
→ Обращения, % автоматизации, средняя оценка, топ причин эскалации, рекомендации
```

Инструменты админ-бота:

| Инструмент | Назначение |
|---|---|
| `admin_get_faq` | Показать записи FAQ |
| `admin_update_faq` | Изменить/добавить/удалить FAQ |
| `admin_get_stats` | Статистика за период |
| `admin_get_escalations` | Список эскалаций с причинами |
| `admin_update_product` | Обновить товар |
| `admin_update_rules` | Обновить бизнес-правила |
| `admin_get_ai_log` | Лог ответов AI |
| `admin_export_data` | Выгрузить данные |

## 10. Решения и выводы

### Принятые решения

| # | Решение | Обоснование |
|---|---|---|
| 1 | Агентный подход (tool use + agentic loop), а не «всё в промпт» | 150 товаров в промпт = $0.06/сообщение, путаница. Tools = $0.005, точность. |
| 2 | 8-10 универсальных инструментов, не 20 узких | AI сам знает что постное/аллергенное. Ему нужны данные, а не логика. |
| 3 | Классификация отдельных сообщений (не сессий) на фазе 2 | Фиксируем обращения, не решения. Сессии — на фазе 5. |
| 4 | Фильтрация мусора кодом, не нейронкой | Дёшево. 1 сообщение, только кнопки, < 3 символов — код справится. |
| 5 | Три режима: off / suggest / auto | Безопасный старт в `suggest`, переход в `auto` по мере доверия. |
| 6 | Self-learning через 3 петли | Система улучшается без ручного вмешательства. |
| 7 | Админ-режим через чат | Клиент управляет без технических знаний. |
| 8 | pgvector для семантического поиска FAQ | SQL-нативный поиск по смыслу, дешевле отдельного vector DB. |
