# Жизненный цикл Системного поста (Пост-трекер)

**Ключевой файл:** `backend_python/services/post_tracker_service.py`

Этот фоновый сервис является "двигателем" для системных публикаций. Он запускается в отдельном потоке при старте бэкенда и работает непрерывно.

---

## 1. Общий цикл работы

Каждые **50 секунд** трекер выполняет задачу публикации (`_publication_check`).

> **Примечание (release2):** Ранее трекер также выполнял _verification_check — двухфазную верификацию через `wall.get`. Эта задача полностью удалена: VK API при успешной публикации гарантированно возвращает `post_id`, промежуточные статусы `publishing` и `possible_error` больше не используются.

## 2. Задача: Публикация (`_publication_check`)

-   **Цель:** Найти и опубликовать посты, время которых пришло.
-   **Пошаговая логика:**
    1.  **Поиск:** Находит в таблице `system_posts` все записи со статусом `pending_publication`, у которых `publication_date` меньше или равно текущему времени.
    2.  **Подстановка переменных:** Вызывает `global_variable_service.substitute_global_variables` для замены плейсхолдеров `{global_...}` на реальные значения.
    3.  **Публикация:** Вызывает сервис `post_service.publish_now`, который отправляет пост в VK.
    4.  **Первый комментарий (release3):** Если поле `first_comment_text` заполнено, после публикации (`wall.post` → получен `vk_post_id`) сервис отправляет комментарий через `wall.createComment` с текстом из `first_comment_text`. Ошибки первого комментария логируются, но не блокируют основную публикацию.
    5.  **Результат:**
        -   **Успех:** VK API возвращает `post_id`. Далее:
            -   Если пост циклический (`is_cyclic = true`), вызывается `_create_next_cyclic_post()` для создания следующей копии (логика описана ниже).
            -   Текущая запись **удаляется** из `system_posts` через `crud.delete_system_post()`.
        -   **Ошибка:** Если публикация не удалась, статус поста меняется на `error`.
    6.  **Уведомление:** В любом случае вызывает `update_tracker.add_updated_project`, чтобы фронтенд узнал об изменении.

## 3. Логика цикличности (`_create_next_cyclic_post`)

-   Проверяет флаг `is_cyclic`. Если `true`:
    -   Вычисляет следующую дату публикации (`_calculate_next_occurrence`), используя `recurrence_interval` и `recurrence_type`.
    -   **Специальная логика для месяцев:** Функция `_add_months` учитывает `recurrence_is_last_day` и `recurrence_fixed_day` для корректного переноса дат (31 января → 28 февраля).
    -   **Проверка лимитов:** Если `recurrence_end_type == 'count'` — уменьшает счётчик. Если `'date'` — проверяет предел.
    -   **Создание:** Если условия позволяют, создаёт **новую запись** в `system_posts` со статусом `pending_publication`, новой датой и унаследованными настройками (включая `first_comment_text`).
    -   **Уведомление:** При создании новой записи вызывается `update_tracker.add_updated_project`.

## 4. Статусы системного поста

| Статус | Описание |
|--------|----------|
| `pending_publication` | Ожидает публикации (трекер подхватит при наступлении даты) |
| `error` | Публикация не удалась (ошибка VK API) |

> **Удалённые статусы (release2):** `publishing` (промежуточный) и `possible_error` (таймаут верификации) — больше не используются.
