# 6. Фоновые системы: VK Callback, трекер обновлений и автоматизации

Этот документ описывает, как бэкенд обрабатывает асинхронные события от VK, как он сообщает фронтенду о необходимости обновить данные, и как работают фоновые автоматизации.

## 1. VK Callback API

VK Callback API позволяет ВКонтакте отправлять на наш сервер уведомления о событиях, происходящих в сообществах (например, "добавлен новый отложенный пост").

### `handle_vk_callback(db, request)` в `services/vk_callback_service.py`

Это единственная функция, которая обрабатывает все входящие колбэки.

-   **Эндпоинт**: `POST /api/callback` (определен в `main.py`).
-   **Поток работы**:
    1.  **Проверка типа события**: Функция смотрит на поле `type` в запросе.
    2.  **Confirmation (Подтверждение сервера)**:
        -   Если `type` равен `confirmation`, это значит, что VK проверяет доступность нашего сервера.
        -   Функция находит проект по `group_id`, извлекает из него `vk_confirmation_code` (который вы вводите в настройках VK и проекта) и возвращает его в теле ответа. Это обязательный шаг для подключения колбэков.
    3.  **Обработка событий с контентом**:
        -   Для событий `wall_post_new`, `postponed_new`, `postponed_delete` и т.д.
        -   Функция определяет, к какому проекту относится событие, по `group_id`.
        -   **Ключевая логика**: Вместо того чтобы пытаться обработать данные из самого колбэка (которые могут быть неполными), функция запускает **полное обновление** соответствующего типа контента. Например, при событии `postponed_new` она вызывает `refresh_scheduled_posts(...)`. Это гарантирует 100% консистентность данных.
        -   После успешного обновления данных в нашей БД, она вызывает `update_tracker.add_updated_project(project.id)`.
    4.  **Ответ VK**: Вне зависимости от результата, функция всегда возвращает `ok` в теле ответа. Это сообщает VK, что уведомление получено, и он не будет пытаться отправить его снова.

## 2. Пост-трекер (Фоновый публикатор)

Это фоновый сервис (`services/post_tracker_service.py`), который запускается при старте бэкенда и работает непрерывно. Выполняет публикацию системных постов по расписанию.

> **Важно (изменение от release2):** Двухфазная верификация публикации (`_verification_check`) была полностью удалена. VK API при успешной публикации возвращает `post_id` — это гарантия, что пост на стене. Промежуточный статус `publishing` и `possible_error` больше не используются.

### Задача: Публикация по расписанию (`_publication_check`)

-   **Триггер**: Каждые ~50 секунд.
-   **Логика**:
    1.  Находит в таблице `system_posts` все посты со статусом `pending_publication`, у которых `publication_date` уже наступило.
    2.  Для каждого такого поста:
        -   **Подстановка переменных:** Вызывает `global_variable_service.substitute_global_variables`.
        -   Вызывает сервис немедленной публикации `post_service.publish_now`.
        -   Если VK API возвращает ID опубликованного поста — ID сохраняется в `vk_post_id`.
        -   **Первый комментарий (release3):** Если поле `first_comment_text` заполнено, после публикации поста автоматически вызывается `wall.createComment` от имени сообщества (`from_group=1`). Глобальные переменные подставляются и в текст комментария.
        -   **Циклические посты:** Если пост отмечен как `is_cyclic`, после успешной публикации создаётся новая запись в `system_posts` с вычисленной датой следующей публикации (`_calculate_next_occurrence`). Поле `first_comment_text` наследуется.
        -   **Очистка:** Текущий (опубликованный) пост удаляется из таблицы `system_posts` сразу после публикации.
        -   Если происходит ошибка — статус меняется на `error`.
    3.  Уведомление через `update_tracker.add_updated_project`.

## 3. Трекер обновлений (`services/update_tracker.py`)

Эта простая система служит мостом между асинхронными колбэками и синхронными запросами от фронтенда.

-   **Проблема**: Когда VK присылает колбэк, бэкенд не может "позвонить" фронтенду и сказать: "Эй, обнови данные!". Фронтенд сам должен спросить.
-   **Решение**: Трекер обновлений.

### `UPDATED_PROJECT_IDS: Set[str]`
-   Это глобальное `set` (множество), которое хранится в памяти бэкенда. Оно содержит ID всех проектов, которые были обновлены через колбэки.

### `add_updated_project(project_id)`
-   **Назначение**: Добавить ID проекта в `UPDATED_PROJECT_IDS`.
-   **Когда вызывается**: Вызывается функцией `handle_vk_callback` или `post_tracker_service` после того, как данные в БД были успешно обновлены.

### `get_and_clear_updates()`
-   **Назначение**: Вернуть список ID всех обновленных проектов и **очистить** `UPDATED_PROJECT_IDS`.
-   **Когда вызывается**: Вызывается эндпоинтом `/api/getUpdates`, который фронтенд опрашивает каждые 5 секунд.

### Общий цикл работы системы

1.  Пользователь создает отложенный пост прямо в интерфейсе VK.
2.  VK отправляет `POST` запрос (`postponed_new`) на эндпоинт `/api/callback` нашего бэкенда.
3.  `handle_vk_callback` получает запрос, вызывает `refresh_scheduled_posts` для обновления кеша в БД.
4.  После успешного обновления, `handle_vk_callback` вызывает `update_tracker.add_updated_project('id-проекта')`. `UPDATED_PROJECT_IDS` теперь содержит `{'id-проекта'}`.
5.  В течение следующих 5 секунд, фронтенд делает фоновый `POST` запрос на `/api/getUpdates`.
6.  `get_and_clear_updates()` возвращает `{"updatedProjectIds": ["id-проекта"]}` и очищает `UPDATED_PROJECT_IDS`.
7.  Фронтенд получает этот ответ и рисует синюю точку рядом с нужным проектом, сообщая пользователю о наличии обновлений.

## 4. Динамический пул воркеров (`services/worker_pool.py`)

Система асинхронной обработки задач с автоматическим масштабированием. Используется для массового редактирования и других ресурсоёмких операций.

### Класс `DynamicWorkerPool`

Пул воркеров, который автоматически адаптируется к нагрузке.

**Конфигурация:**
| Параметр | По умолчанию | Описание |
|----------|--------------|----------|
| `min_workers` | 2 | Минимум параллельных задач |
| `max_workers` | 20 | Максимум параллельных задач |
| `initial_workers` | 5 | Начальное количество |
| `scale_up_threshold` | 0.8 | Нагрузка > 80% → масштабирование вверх |
| `scale_down_threshold` | 0.3 | Нагрузка < 30% → масштабирование вниз |
| `scale_step` | 2 | Шаг изменения |
| `cooldown_seconds` | 3.0 | Пауза между масштабированиями |
| `rate_limit_per_second` | 0 | Лимит запросов/сек (0 = без лимита) |

**Использование:**
```python
pool = DynamicWorkerPool(min_workers=2, max_workers=15, name="bulk_edit")

async def process_post(post):
    async with pool.acquire():
        await edit_post(post)

# Параллельная обработка всех постов
await asyncio.gather(*[process_post(p) for p in posts])
```

### Логика масштабирования

1. **Scale Up**: Когда загрузка (active / current) > 80%, добавляем воркеров
2. **Scale Down**: Когда загрузка < 30% и прошло `cooldown_seconds`, уменьшаем
3. **Cooldown**: Между масштабированиями обязательная пауза для стабильности

### Предустановленные пулы

В `worker_pool.py` определены готовые пулы:

```python
# Для массового редактирования
bulk_edit_pool = DynamicWorkerPool(
    min_workers=2,
    max_workers=15,
    initial_workers=5,
    rate_limit_per_second=3,  # Не более 3 запросов к VK API в секунду
    name="bulk_edit"
)
```

### Статистика пулов

Доступна через API `GET /api/bulkEdit/workers/stats`:

```json
{
  "bulk_edit": {
    "currentWorkers": 5,
    "activeWorkers": 3,
    "loadPercent": 60,
    "completedTasks": 150,
    "failedTasks": 2,
    "avgTaskTimeMs": 1234.56,
    "scaleUps": 10,
    "scaleDowns": 5
  }
}
```

### Task Monitor (`services/task_monitor.py`)

Система отслеживания долгих фоновых задач.

**Функции:**
| Функция | Описание |
|---------|----------|
| `start_task(task_id, task_type, description)` | Регистрация задачи |
| `update_progress(task_id, completed, total, current)` | Обновление прогресса |
| `complete_task(task_id, result)` | Пометить как выполненную |
| `fail_task(task_id, error)` | Пометить как провалённую |
| `get_task_status(task_id)` | Получить текущий статус |

**Retry-логика (release.md):** Все функции обёрнуты в `_db_operation_with_retry()` — 3 попытки с экспоненциальной задержкой (0.5с → 1с → 2с). Критично для PostgreSQL в облаке, где idle-соединения могут закрываться.

## 5. Автоматизация историй (Stories Automation)

**Ключевые файлы:** `services/automations/stories/core.py`, `logic.py`, `stats.py`, `viewers.py`, `retrieval.py`

Фоновый сервис автоматической публикации историй из постов VK. Запускается через APScheduler при старте каждого Gunicorn-воркера.

### Защита от дублирования (release3 — трёхуровневая)

Dockerfile запускает `gunicorn -w 4`, и каждый воркер создаёт собственный экземпляр APScheduler. Без защиты все 4 воркера опубликуют одну и ту же историю.

| Уровень | Механизм | Файл |
|---------|----------|------|
| 1. **Redis-лок** | Ключ `vk_planner:story_publish:{project_id}:{post_id}` с TTL 5 мин. Только один воркер пройдёт. | `core.py` |
| 2. **UNIQUE constraint** | `UniqueConstraint('project_id', 'vk_post_id')` в таблице `stories_automation_logs`. БД не позволит вставить дубль. | `models_library/automations.py` |
| 3. **IntegrityError** | При `db.commit()` перехватывается `IntegrityError` — транзакция откатывается без краша. | `logic.py` |

### Финализация статистики (release3)

Архивные истории, чья статистика больше не меняется, помечаются `stats_finalized=True`. При следующем сборе статистики такие истории пропускаются.

**Правила финализации:**
- Для архивной истории запрашивается статистика из VK API.
- Если все метрики (views, replies, shares, etc.) совпадают с текущими значениями в БД → `stats_finalized = True`.
- При VK API Error 15 (Access denied) или Error 100 (Story deleted) для архивных историй → `stats_finalized = True` (permanently inaccessible).
- При реактивации истории (`is_active` переходит из false в true) флаг сбрасывается.
- Результат: проект с 176 историями вместо ~352 запросов делает ~4-6 (только свежие). Экономия ~98% запросов.

## 6. Батчевая загрузка данных (release3)

**Ключевые файлы:** `crud/batch_crud.py`, `routers/batch.py`, `services/api/batch.api.ts`

### Проблема (release3)
Старый подход: эндпоинт `/getAllPostsForProjects` загружал все 7 типов данных одним запросом. При 145 проектах ответ превышал лимит Yandex Cloud Serverless Containers (~3.5 МБ), возвращая 502 Bad Gateway.

### Новый подход

**6 батчевых POST-эндпоинтов** (принимают `{ project_ids: string[] }`):

| Эндпоинт | CRUD-функция | Описание |
|----------|-------------|----------|
| `POST /api/getPostsBatch` | `get_posts_batch()` | Опубликованные посты |
| `POST /api/getScheduledPostsBatch` | `get_scheduled_posts_batch()` | Отложенные посты |
| `POST /api/getSuggestedPostsBatch` | `get_suggested_posts_batch()` | Предложенные посты |
| `POST /api/getSystemPostsBatch` | `get_system_posts_batch()` | Системные посты |
| `POST /api/getNotesBatch` | `get_notes_batch()` | Заметки |
| `POST /api/getStoriesBatch` | `get_stories_batch()` | Истории (без поля `viewers` для уменьшения размера) |

### Чанкинг на фронтенде (`useDataInitialization.ts`)

Двухфазная загрузка:
1. **Phase 1:** Загрузка проектов + счётчиков → UI готов к взаимодействию.
2. **Phase 2:** Параллельная загрузка контента через `Promise.allSettled`.

Конфигурация чанкинга:
- `HEAVY_CHUNK_SIZE = 10` — посты и истории загружаются порциями по 10 проектов.
- `MAX_CONCURRENT = 5` — ограничение параллельных запросов.
- Лёгкие типы (scheduled, suggested, system, notes) загружаются одним запросом.

### Обнаруженные лимиты Yandex Cloud
- Ограничение размера HTTP-ответа: ~3.5 МБ (22 проекта = 3.06 МБ ✅, 24 проекта = 502 ❌).
- `MAX_RETRIES` в `apiClient.ts` увеличен с 1 до 3 (экспоненциальный бэкофф 1с → 2с → 4с).

## 7. Управление DB-сессиями для фоновых задач (release.md)

### Миграция на `scoped_session`

**Проблема:** Обычный `sessionmaker` создаёт сессии, которые не потокобезопасны. При параллельной обработке (ThreadPoolExecutor, BackgroundTasks) несколько потоков использовали одну сессию.

**Решение (`database.py`):**
- `SessionLocal = scoped_session(sessionmaker(...))` — каждый поток получает свою изолированную сессию.
- `get_db()` использует `SessionLocal.remove()` для корректной очистки (вместо `db.close()`).
- Хелпер `get_background_session()` для фоновых задач.
- Конфигурация пула: `pool_pre_ping=True`, `pool_recycle=120`, `pool_size=10`, `max_overflow=20`.

### Замена `db.close()` на `SessionLocal.remove()`
Во всех файлах, работающих с сессиями вне контекста FastAPI Depends:
- `routers/market.py` — удалена локальная `get_db()`, импортируется корректная версия.
- `services/vk_api/token_manager.py` — замена во всех функциях.
- `services/vk_api/admin_tokens.py` — замена во всех функциях.
- `services/token_log_service.py` — замена `SessionLocal()` на `_session_factory()` (независимая сессия).
